## Trade-offs

The v1.1 milestone surfaced three architectural trade-offs worth documenting
honestly. Each is a recorded decision, not a slip in discipline.

### 1. `v1.0.0 → v1.0.1` freeze-day-after re-tag

`llm-agent-rag v1.0.0` was cut **2026-05-21** as the frozen public-API point
(the v1.0 milestone artifact). `v1.0.1` was cut **2026-05-22 — the day
after** — for the back-edge refresh so the strict dep-currency gate could
fire green across all 5 repos.

**KE-2 holds.** The back-edge bump moves no exported symbol: the
`internal/apisnapshot` exported-surface gate at `v1.0.1` is byte-identical
to the `api/v1.snapshot.txt` baseline from `v1.0.0`. `v1.0.1` is a **patch
tag with chore-only semantics within `v1.x`** — the back-edge `require
llm-agent vX.Y.Z` line in `go.mod` is a build-test artifact (which core
version was rag tested against), not a release contract.

The freeze-day-after timing is documented here so future readers see the
intent: it was not a slip in API discipline. The alternative — leaving
`rag@v1.0.0` pointing at a stale `llm-agent v0.4.0` while the umbrella's
strict-equality gate stares at the back-edge — was rejected because the gate
exists to surface drift; exempting the back-edge with a global "skip rag"
would weaken the gate against every future drift. Trade-off §3 documents the
narrower exemption actually chosen.

### 2. Topological-order miss in the cascade (Wave 4 → Wave 6 correction)

The Phase 33 cascade plan ordered repos as `rag → core → otel → cs →
providers` — intuitively "leaves first". But the correct topological order
against the actual dep DAG is `rag → core → providers → otel → cs`:
`customer-support` is the sink (it pins all four other repos);
`providers` is an internal node pinned by `cs`. Cutting `cs v0.2.1` in
Wave 4 before `providers v0.2.1` existed in Wave 5 meant `cs`'s
`providers` pin lagged — Wave 6 was added mid-flight to cut `cs v0.2.2`
once `providers v0.2.1` was in place.

**Future cascades should sort by `tsort` against the dep DAG**, not by
intuition. The 9-slice expansion of Phase 34 (originally 3 slices) is the
visible cost of the miss; the correction was bounded and the strict gate
caught it before the milestone closed (the gate exit-0 in Wave 8 is the
proof the cascade is now in topological order at the tag-set level).

### 3. `rag ↔ core` cycle exemption in the dep-currency gate

The dep DAG is **not a DAG** at the tag level — it is a diamond with a
back-edge. `core` pins `rag` (the public-facade module dep);
`rag` pins `core` (so rag's test suite can run against a real core).
Strict-equality currency across a cycle is mathematically unsatisfiable —
every re-tag of one side stales the other forever.

**Wave 7 ships exactly one auditable exemption** in
`scripts/dep-currency-check.sh`:

```bash
# CYCLE EXEMPTION (KE-2 corollary, operator-confirmed 2026-05-20):
# rag↔core is a dep cycle (rag pins core via the back-edge to test
# against; core pins rag via the public-facade module dep). Strict
# equality is mathematically unsatisfiable across a cycle — each
# re-tag of one side stales the other forever. rag's pin of core is
# a build artifact ("which core version was rag tested against?"),
# not a release contract; currency between consumers of rag is what
# matters and is checked on every other edge. This is the ONE
# explicit, auditable exemption.
if [ "$sister" = "llm-agent-rag" ] && [ "$r" = "llm-agent" ]; then
  echo "SKIP: rag back-edge to core (cycle exemption — KE-2 corollary)"
  continue
fi
```

Every other edge in the dep DAG is strict-checked. Future maintainers
considering broadening the exemption should consult this audit + the script
comment + KE-2 first — the exemption is narrow on purpose.

### Decision-1 sub-tradeoff (carried over from research)

The gate is **core-PR-scoped** (lives in `llm-agent`'s `umbrella.yml`).
Sister-repo PRs are not gated individually for cross-sibling currency at
their own CI; staleness on a sister-repo PR is caught when the next umbrella
build runs in the core repo, not in real time on the sister PR. A
per-sister-repo gate is plausible future work but is explicitly out of scope
for v1.1 (KE-1 — alignment, not features).
