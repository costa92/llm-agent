# Research Summary: v0.8 GraphRAG Tier-3 — Communities, Global Search & Fuzzy Resolution

**Target:** `llm-agent-rag` (`github.com/costa92/llm-agent-rag`), standalone Go RAG SDK at tag `v0.4.0`
**Milestone direction:** GraphRAG Tier-3 — cluster the entity graph into hierarchical communities, summarize them, answer whole-corpus "sense-making" questions by map-reduce over those summaries, and merge near-duplicate entities by embedding similarity.
**Researched:** 2026-05-19
**Overall confidence:** HIGH on seam fit and the stdlib-clustering question (grounded in the v0.4.0 codebase scan + well-established algorithms); HIGH on the Lazy-vs-full keystone (Microsoft's own published cost data); MEDIUM on cost/quality numbers (vendor blogs, directionally consistent); MEDIUM on the exact phase slice split.

---

## Executive Summary

v0.7 shipped Tier-1 lightweight GraphRAG: an `EntityExtractor` seam, a `store.GraphStore` optional capability (stdlib in-memory + `postgres` recursive-CTE), and a `retrieve.GraphRetriever` that fuses neighborhood-expansion hits as a fourth RRF signal. It deliberately stopped at the LightRAG end of the spectrum and named the deferral list precisely: **Leiden / hierarchical community detection, LLM community summaries, map-reduce global search, DRIFT search, and embedding-similarity fuzzy entity resolution**. v0.8 GraphRAG Tier-3 is the explicit continuation that delivers that list.

The shape of the v0.8 work is different from v0.7 in one structural way that drives every keystone below: **Tier-3 is not another retriever.** v0.7's whole milestone fit under `retrieve.Retriever` because neighborhood expansion *is* a retrieval — it returns `[]store.Hit` to fuse by RRF. Global search is a fundamentally different answer path: it does not rank chunks, it maps an LLM over community *summaries* and reduces partial answers into one. It cannot implement `retrieve.Retriever` honestly. So v0.8 adds a **second answer path alongside `rag.System.Ask`** (`AskGlobal`, or an `Ask` mode switch), not a new `Retriever`. This is the single most important framing for the roadmapper.

The keystone decision is **full Tier-3 community summarization vs LazyGraphRAG**. Microsoft's own published numbers make this nearly unambiguous: full MS-GraphRAG summarizes every community in the hierarchy up front — a second large LLM pass that, on Microsoft's own reporting, dominates indexing cost — while **LazyGraphRAG defers summarization to query time and runs indexing at roughly 0.1% of full-GraphRAG cost** while matching or beating it on answer quality. For a solo, stdlib-disciplined SDK whose v0.7 keystone KG-3 already mandates a zero-LLM default path, paying a whole-corpus LLM pass at *ingest* time — on every re-ingest — is the wrong default. **v0.8 should target the LazyGraphRAG model: detect communities deterministically at ingest (cheap, no LLM), and generate community reports lazily at query time, cached.** Full eager summarization stays available as an opt-in but is not the default and not the thing the milestone is built around.

Community detection itself is the good news: **it is pure-stdlib-implementable and store-agnostic.** Label propagation and Louvain are a few hundred lines of stdlib Go over the adjacency structure the in-memory `GraphStore` already builds in `Neighborhood`. Postgres has no built-in Leiden — and it does not need one: detection runs *in Go* over the graph regardless of which store backs it (load entities/relations, cluster in memory, write community IDs back). This keeps the in-memory default pure stdlib (hard rule honored) and means **zero new dependencies for the entire milestone**, exactly like v0.7.

The honest scope call: **DRIFT search should itself be deferred past v0.8.** DRIFT is a refinement *on top of* working global + local search — it is the hybrid that runs a local follow-up loop seeded by a global primer. Building global search, lazy community reports, and fuzzy resolution is already a full milestone. DRIFT is a believable v0.9 once global and local both exist and are evaluated. v0.8 should ship global + local cleanly and name DRIFT as the deferral, exactly as v0.7 named Tier-3.

## 2. The GraphRAG Spectrum — where v0.8 lands (continued from v0.7 §2)

| Tier | Construction | Retrieval / answer path | Cost / update | v0.8 disposition |
|---|---|---|---|---|
| **1. Lightweight** (shipped v0.7) | LLM/dict extraction; exact-match canonicalize | entity-link → neighborhood expansion → RRF fusion | one LLM call/chunk; cheap union-merge updates | **Done.** |
| **2. Path/subgraph** | tier 1 + typed relations | path ranking, subgraph-as-evidence | same extraction cost | Still deferred — orthogonal to Tier-3, a v0.9+ item. |
| **3a. Full MS-GraphRAG** | tier 1 + hierarchical community detection + **eager** LLM community summaries | global search (map-reduce over reports) + local search + DRIFT | community summarization is a **second whole-corpus LLM pass**; slow refresh | **Partially in scope** — detection + global search yes; *eager* summarization is opt-in, not default. |
| **3b. LazyGraphRAG** | tier 1 + hierarchical community detection; **no** up-front summaries | global search defers report generation to **query time**, cached; best-first community selection | indexing ≈ **0.1%** of full-GraphRAG cost; updates only re-detect, never re-summarize | **← RECOMMENDED v0.8 default.** |
| **3c. DRIFT** | 3a/3b | global *primer* → local follow-up loop | extra query-time LLM calls | **Deferred to v0.9.** Refinement on top of working global+local. |

**Where v0.8 sits: Tier-3b (LazyGraphRAG) as the default, with eager summarization (3a) as an opt-in.** Rationale:

1. **Microsoft's own cost data.** LazyGraphRAG was published by the GraphRAG team specifically because full-GraphRAG indexing cost — dominated by the community-summary pass — is a real adoption barrier. Reported indexing cost is ~0.1% of full GraphRAG at comparable or better answer quality. An SDK whose v0.7 keystone KG-3 already insists on a zero-LLM default path should not make a whole-corpus LLM pass the indexing default.
2. **It respects the re-ingest story.** v0.7's KG-5 made incremental graph reconciliation a day-one requirement. Eager summaries make re-ingest expensive: change one document, and every community it touches must be re-summarized. Lazy reports are invalidated cheaply (drop the cache entry) and regenerated only if queried.
3. **It is the smaller, safer, still-complete milestone.** Detection + lazy global search + fuzzy resolution is a coherent, shippable Tier-3 slice. Eager summarization is then a thin opt-in (`PrewarmCommunityReports`) on top — not a separate architecture.
4. **Determinism.** Detection is deterministic (stdlib, fixed seed/tie-breaking). Lazy report generation is LLM-backed but scripted-model tested. The lazy model keeps the deterministic and non-deterministic halves cleanly separable.

**Explicitly deferred past v0.8:** DRIFT search; path-ranking / subgraph-as-evidence; claim/covariate extraction; any graph database.

## 3. Community Detection — Algorithm & the Dependency Question

**Question 1 answered: community detection is pure-stdlib-implementable, and it runs in Go regardless of store.**

The algorithm operates on an adjacency structure — `map[entityID][]neighbor` with edge weights. The in-memory store already builds exactly this in `store/graph.go:104-111`; the postgres store can return entities+relations to build it in memory. **Postgres has no built-in Leiden/Louvain and does not need one** — detection is a Go computation over the loaded graph, and the store only persists the resulting `(entity → communityID)` assignment. This is store-agnostic by construction.

| Algorithm | Quality | Complexity to implement in stdlib Go | Determinism | Recommendation |
|---|---|---|---|---|
| **Label propagation (LPA)** | Good for clear community structure; can be unstable | ~150 lines; iterative neighbor-label voting | Needs fixed iteration order + deterministic tie-break; otherwise non-deterministic | Good *fallback* / fastest path |
| **Louvain** | Strong modularity optimization; the de-facto standard before Leiden | ~300–400 lines; modularity-gain greedy moves + graph coarsening for hierarchy | Deterministic with fixed node iteration order + tie-break rule | **← RECOMMENDED.** Best quality:effort ratio in stdlib; native hierarchy via coarsening passes |
| **Leiden** | Best quality; fixes Louvain's badly-connected-community defect | ~600+ lines; adds a refinement phase + well-connectedness guarantees | Deterministic with care | **Not v0.8.** Marginal quality gain over Louvain for this SDK's graph sizes does not justify the implementation/test surface. Document as a future swap behind the seam. |

**Recommendation: implement Louvain in pure stdlib.** Louvain naturally produces the **hierarchy** Tier-3 needs: each coarsening pass yields a level (level 0 = finest communities, higher levels = communities-of-communities). The output is `[]Community` where each `Community` has a `Level`, a parent ID, member entity IDs, and (at level 0) member relations. Label propagation ships as a faster, simpler alternative behind the same `CommunityDetector` seam — mirroring v0.7's `LLMEntityExtractor` vs `DictionaryEntityExtractor` dual-mode pattern.

**Determinism is non-negotiable (KG3-6).** Louvain's result depends on node visit order and tie-breaking. v0.8's detector must: iterate entities in sorted `ID` order, break modularity-gain ties by lowest community ID, and use no randomness (no random restarts). A given graph then always yields the same community hierarchy — unit-testable with golden assertions, exactly as `Canonicalize` is today.

**Seam:**
```go
package graph

// Community is one cluster in the hierarchy. Level 0 is the finest;
// higher levels group lower-level communities.
type Community struct {
    ID        string
    Level     int
    ParentID  string   // "" at the top level
    EntityIDs []string // members (sorted)
    RelationIDs []string
}

// CommunityDetector partitions a Graph into a community hierarchy.
// Implementations are deterministic — same graph, same hierarchy.
type CommunityDetector interface {
    Detect(ctx context.Context, g Graph) ([]Community, error)
}
// LouvainDetector{Resolution float64}  — recommended default
// LabelPropagationDetector{}           — faster, simpler alternative
```
No LLM, no embedder, no new dependency — `CommunityDetector` is a pure-stdlib computation seam.

## 4. Community Summaries — Lazy vs Eager (the keystone, see KG3-2)

A **community report** is an LLM-written summary of one community: a title, a paragraph synopsis, and the key entities/relations/claims, generated from the community's member entity descriptions and relation descriptions.

- **Eager (full MS-GraphRAG):** every community in the hierarchy is summarized at ingest. A second whole-corpus LLM pass. Expensive; re-ingest re-summarizes every touched community.
- **Lazy (LazyGraphRAG):** no summaries at ingest. At query time, global search selects the relevant communities and generates reports for just those, on demand, cached. Microsoft reports indexing at ~0.1% of full-GraphRAG cost.

**Recommendation: Lazy by default.** The `CommunitySummarizer` seam is LLM-backed (`generate.Model`, scripted-model tested). The `rag.System` holds a **community-report cache** keyed by community ID + a content hash of the community (so a re-detected community with the same membership reuses its cached report; a changed one misses and regenerates). Eager mode is a thin opt-in: a `System.PrewarmCommunityReports(ctx, namespace)` method that walks the hierarchy and fills the cache — same summarizer, same cache, just called eagerly.

```go
package graph

// CommunityReport is an LLM-written summary of one community.
type CommunityReport struct {
    CommunityID string
    Title       string
    Summary     string
    ContentHash string // hash of the community's membership; cache key
}

type CommunitySummarizer interface {
    Summarize(ctx context.Context, c Community, g Graph) (CommunityReport, error)
}
// LLMCommunitySummarizer{Model generate.Model}  — scripted-model tested
```

Determinism: the summarizer is LLM-backed and therefore non-deterministic in production, but unit-tested against a scripted model returning a canned report; the *cache key* (content hash) is deterministic.

## 5. Global Search — the Map-Reduce Answer Path

**Question 3 answered: global search is NOT a `retrieve.Retriever`.** It does not rank or return `[]store.Hit`. It answers whole-corpus "sense-making" questions ("what are the main themes across all documents?") by:

1. **Select communities.** Choose a level of the hierarchy (or, LazyGraphRAG-style, best-first select the most relevant communities by a cheap relevance signal — e.g. embedding the query against community member-name text). Lazy: generate reports for the selected communities now (cache-filled).
2. **Map.** For each selected community report, prompt the `generate.Model`: "given this community summary, what does it contribute to answering Q?" → a partial answer + a self-rated helpfulness score.
3. **Reduce.** Rank partial answers by score, drop zero-score ones, and prompt the model once more to synthesize the survivors into a final answer.

**Where it slots:** a new method on `rag.System`:
```go
func (s *System) AskGlobal(ctx context.Context, question string, opts GlobalOptions) (Answer, error)
```
It reuses the existing `Answer` struct, the `obs.Counter` cost-tracking shell, and `Diagnostics` (with a new global block: communities consulted, per-community map scores, map/reduce token counts). It does **not** go through `s.retrieve`, `HybridRetriever`, the reranker, or the packer — those are chunk-retrieval machinery and global search is a different shape. `Ask` (local/hybrid) is unchanged.

This mirrors how v0.7 kept `MultiHopRetriever` and `GraphRetriever` as *distinct* things rather than forcing one into the other — the repo already has the discipline of "different shape → different type, not a flag."

**Local search clarification:** v0.7's `GraphRetriever` already *is* Tier-3 local (entity-anchored) search. v0.8 local search = `GraphRetriever` optionally enriched with the community reports of the entities it reached. That enrichment is a small additive change to `GraphRetriever` (or a wrapping decorator), not a new path.

## 6. Fuzzy / Embedding-Similarity Entity Resolution

**Question 5 answered.** v0.7's `Canonicalize` merges only exact `(normalized name, type)` matches — "Acme" and "Acme Corp" stay two nodes. v0.8 adds embedding-similarity merge over the `embed.Embedder` seam.

**Design — a pre-pass before `Canonicalize`, not a rewrite of it:**
1. Embed each raw entity's `name` (optionally `name + ": " + description`) with the `embed.Embedder`.
2. Cluster by cosine similarity above a threshold, restricted to same-`Type` entities (don't merge a person with an org), using `embed.CosineSimilarity` (already in `embed/hash.go:33`).
3. Within each cluster, pick a canonical surface form deterministically (e.g. longest name, tie-broken lexically) and rewrite every member's `Name` to it.
4. Feed the rewritten entities into the existing `graph.Canonicalize` — which now exact-merges them because their names agree.

This keeps `Canonicalize` (and all its v0.7 tests) untouched; fuzzy resolution is a clean additive stage that *normalizes input to* `Canonicalize`.

**Seam:**
```go
package graph

// EntityResolver rewrites near-duplicate entity names to a shared
// canonical surface form before Canonicalize runs.
type EntityResolver interface {
    Resolve(ctx context.Context, entities []Entity) ([]Entity, error)
}
// EmbeddingEntityResolver{Embedder embed.Embedder; Threshold float64}
// NoopEntityResolver{} — the default; v0.7 behavior unchanged
```

**Determinism / test strategy (KG3-6):** the resolver is embedding-backed → unit-tested against a **scripted embedder** returning fixed vectors, asserting that "Acme"/"Acme Corp" collapse and unrelated entities do not. Clustering must be deterministic: sorted iteration, deterministic canonical-form pick, fixed threshold. The default `rag.System` keeps `NoopEntityResolver` so existing behavior is byte-identical unless fuzzy resolution is explicitly wired — same opt-in discipline as v0.7's KG-4.

**Honest limitation to document:** embedding-similarity resolution has false positives ("Apple" the company vs "Apple" the fruit can embed close); the threshold is a tuning knob and aggressive merging silently corrupts the graph. Ship it conservative (high threshold), opt-in, and documented.

## 7. Determinism, Testing & Eval

- **Community detection** — deterministic by construction (KG3-6): sorted node order, deterministic tie-breaks, no randomness. Golden-output unit tests like `Canonicalize`'s.
- **Community summarizer** — LLM-backed → scripted-`generate.Model` tests covering canned reports and malformed output (lenient parsing, per v0.7 precedent).
- **Fuzzy resolver** — embedding-backed → scripted-`embed.Embedder` tests with fixed vectors.
- **Global search** — the map and reduce steps are LLM-backed → scripted-model tests asserting community selection, per-community map scoring, and reduce synthesis on a fixed tiny graph.
- **Eval gap (MEDIUM-confidence, flag for the roadmapper):** `eval/graph.go`'s `RunGraphAB` measures *chunk recall* — the right metric for local/hybrid retrieval, the **wrong** metric for global search, which produces a synthesized answer with no single "correct chunk set." Global-search eval needs the **RAG-Triad / `LLMJudge`** path (`eval/triad.go`, `eval/judge.go`) — comprehensiveness/groundedness on whole-corpus questions — not recall@k. v0.8's eval slice should add a global-search Triad harness, not extend `RunGraphAB`.

## 8. Cost & Update Story

| Concern | Eager (3a) | Lazy (3b) — recommended |
|---|---|---|
| Indexing LLM cost | extraction + **whole-corpus summary pass** | extraction only; ~0.1% of eager per Microsoft's reporting |
| Re-ingest cost | re-summarize every touched community | re-detect communities (stdlib, no LLM); drop stale report-cache entries |
| Query cost | cheap (reports pre-made) | map/reduce LLM calls at query time, on selected communities only; cached after first use |
| First-query latency | low | higher on a cold cache for that community set |

**Incremental re-detection (KG3-7):** v0.7's KG-5 made re-ingest reconcile the graph. Communities are derived from the graph, so on any re-ingest that changes entities/relations, **communities must be re-detected** for that namespace. Because detection is cheap stdlib, full re-detection per affected namespace is acceptable for v0.8 — incremental community *maintenance* (updating only changed communities) is a known optimization, explicitly deferred. The report cache is invalidated by content-hash mismatch: re-detected communities with unchanged membership keep their cached reports for free.

## Keystone Decisions (for the roadmapper to ratify — continuing the v0.7 KG-* line)

- **KG3-1 — Spectrum position.** v0.8 targets **Tier-3** GraphRAG: hierarchical community detection + community summaries + map-reduce global search + embedding-similarity fuzzy entity resolution. DRIFT search, path-ranking/subgraph-as-evidence, and claim extraction are explicitly out of scope.
- **KG3-2 — Lazy by default.** Community **report generation is lazy** (LazyGraphRAG model): communities are detected at ingest (cheap, no LLM); reports are generated at query time and cached. Eager whole-corpus summarization is a thin opt-in (`PrewarmCommunityReports`), not the default and not what the milestone is architected around. Rationale: Microsoft's own ~0.1%-indexing-cost data and v0.7's zero-LLM-default-path discipline (KG-3).
- **KG3-3 — Community detection is pure stdlib and store-agnostic.** A `graph.CommunityDetector` seam with a **stdlib Louvain** default (deterministic; native hierarchy via coarsening) and a `LabelPropagationDetector` alternative — dual-mode, mirroring v0.7's KG-3. Detection runs in Go over the loaded graph regardless of store; Postgres only *persists* `(entity → community)` assignments. **No graph database, no new dependency.**
- **KG3-4 — Global search is a separate answer path, not a `Retriever`.** A new `rag.System.AskGlobal` method does map-reduce over community reports. It does not implement `retrieve.Retriever`, does not pass through `HybridRetriever`/rerank/pack, and does not change `Ask`. Local search remains v0.7's `GraphRetriever`, optionally enriched with community reports.
- **KG3-5 — Communities persist behind the `GraphStore` capability.** Community/report storage extends the graph-store capability (additive methods on `GraphStore`, or a sibling `CommunityStore` capability — a Phase-23 plan-time call): stdlib in-memory impl + `postgres` `_communities`/`_community_reports` tables in the existing `Migrate()`. Consumers type-assert and degrade gracefully — the v0.7 `GraphStore`/`LexicalSearcher` contract.
- **KG3-6 — Determinism is mandatory for every new type.** Community detection is deterministic by construction (sorted order, deterministic tie-breaks, no randomness). LLM-backed types (summarizer, global map/reduce) are scripted-`generate.Model` tested; the embedding-backed fuzzy resolver is scripted-`embed.Embedder` tested. Cache keys (community content hashes) are deterministic.
- **KG3-7 — Re-detection on re-ingest from day one.** Re-ingest that changes the graph re-detects communities for that namespace (cheap stdlib); the report cache is invalidated by content-hash mismatch. Full re-detection is acceptable for v0.8; incremental community maintenance is a documented deferred optimization. Continues v0.7's KG-5.
- **KG3-8 — Fuzzy resolution is an opt-in pre-pass.** `EmbeddingEntityResolver` rewrites near-duplicate names to a canonical form *before* `graph.Canonicalize` runs — `Canonicalize` and its tests are untouched. The `rag.System` default is `NoopEntityResolver` (v0.7 behavior byte-identical). Ships conservative (high threshold) and documented as having false-positive risk.

## Proposed Phase Breakdown (v0.8 — global numbering continues; v0.7 ended at Phase 22)

**Three phases, ~2–3 slices each — matching the v0.7 cadence (Phases 20–22).**

### Phase 23 — Community detection & graph-store persistence

**Goal:** detect a deterministic community hierarchy from the canonicalized graph and persist it behind the graph-store capability.

- `23-01` `graph` package: `Community` type + `CommunityDetector` seam; **stdlib Louvain** detector (deterministic — sorted order, deterministic tie-breaks, hierarchy via coarsening passes); golden-output unit tests. `LabelPropagationDetector` as the alternative impl.
- `23-02` community persistence on the graph-store capability (`UpsertCommunities` / `Communities`); pure-stdlib in-memory impl; shared conformance suite (extends `storetest`).
- `23-03` `postgres` community persistence — `_communities` table in `Migrate()`; detection still runs in Go. Community detection wired into `Import` as a post-`Canonicalize` stage; counts surface on `ImportResult.Graph`; re-detection on `ReplaceSource` re-ingest (KG3-7).

### Phase 24 — Community summaries & global search

**Goal:** lazy community reports and a map-reduce global-search answer path.

- `24-01` `graph` package: `CommunityReport` type + `CommunitySummarizer` seam; `LLMCommunitySummarizer` over `generate.Model` (lenient parse, scripted-model tests incl. malformed output). Report cache keyed by community content hash; in-memory + `postgres` `_community_reports` persistence.
- `24-02` `rag.System.AskGlobal`: community selection → map (per-community partial answer + helpfulness score) → reduce (synthesize). Lazy report generation on cache miss. New global block in `Diagnostics`; scripted-model tests on a fixed graph.
- `24-03` opt-in `PrewarmCommunityReports` (eager mode); local search enriched with the community reports of reached entities; deterministic worked example.

### Phase 25 — Fuzzy entity resolution & evaluation

**Goal:** embedding-similarity entity merge and a global-search eval harness.

- `25-01` `graph.EntityResolver` seam + `EmbeddingEntityResolver` (cosine clustering over `embed.Embedder`, same-type-only, deterministic canonical-form pick); scripted-embedder tests; wired as an opt-in pre-pass before `Canonicalize` in `Import`; `NoopEntityResolver` default.
- `25-02` global-search eval: a RAG-Triad / `LLMJudge` harness for whole-corpus questions (comprehensiveness/groundedness — **not** `RunGraphAB` recall@k); deterministic scripted-model eval example.
- `25-03` `docs/graphrag.md` updated — Tier-3 usage, the lazy-vs-eager tradeoff, the fuzzy-resolution false-positive caveat, and the v0.9 deferral list (DRIFT, incremental community maintenance, path-ranking).

## Pitfalls & How v0.8 Pre-empts Them

| Pitfall (from current sources) | How it bites | v0.8 mitigation |
|---|---|---|
| **Eager-summarization cost** — whole-corpus LLM pass dominates indexing | Slow/expensive ingest; re-ingest re-summarizes everything | **KG3-2:** lazy by default; reports generated per-query and cached; eager is opt-in only. |
| **Global search treated as a retriever** — forcing map-reduce under `retrieve.Retriever` | A dishonest interface; rerank/pack machinery applied to non-chunk output | **KG3-4:** `AskGlobal` is a separate path; never implements `Retriever`. |
| **Non-deterministic community detection** — Louvain/LPA depend on node order & random restarts | Flaky golden tests; non-reproducible graphs | **KG3-6:** sorted iteration, deterministic tie-breaks, zero randomness. |
| **Fuzzy-merge false positives** — embedding similarity merges genuinely distinct entities | Silent graph corruption, degraded recall | **KG3-8:** same-type-only, conservative high threshold, opt-in (`Noop` default), documented caveat. |
| **Re-detection forgotten on re-ingest** — communities go stale vs the graph | Global search answers from an outdated structure | **KG3-7:** re-detect per namespace on any graph-changing re-ingest, day one. |
| **Report-cache staleness** — cached report no longer matches its community | Global search cites a community it no longer reflects | Content-hash cache key — membership change ⇒ cache miss ⇒ regenerate. |
| **Scope creep into DRIFT** — DRIFT looks like "just one more search mode" | Milestone bloats; global+local ship half-tested | DRIFT explicitly deferred to v0.9; v0.8 ships global+local cleanly first. |
| **Leiden over-engineering** — chasing best-in-class clustering | 600+ lines of subtle stdlib code + test surface for marginal gain | **KG3-3:** Louvain is the v0.8 default; Leiden documented as a future swap behind the `CommunityDetector` seam. |
| **Wrong eval metric** — running global search through `RunGraphAB` recall@k | Can't tell if global search is any good | §7 / `25-02`: global search evaluated by the RAG-Triad/`LLMJudge` path, not chunk recall. |
| **Postgres-Leiden assumption** — expecting the DB to cluster | Wasted effort hunting a non-existent feature | **KG3-3:** detection runs in Go; Postgres only persists assignments. |

## Gaps / Open Questions for the Roadmapper

- **`GraphStore` extension vs `CommunityStore` sibling.** Whether community persistence is additive methods on `GraphStore` or a new sibling capability interface — a Phase-23 plan-time call. A sibling keeps `GraphStore` small and lets a store implement graph traversal without communities; extending `GraphStore` is fewer types. Lean sibling `CommunityStore`.
- **Community-relevance signal for lazy selection.** LazyGraphRAG uses a best-first community ranking; the cheapest in-repo signal is embedding the query against community member-name text via `embed.Embedder`. Confirm whether v0.8 wants embedding-based selection or a simpler level-pick — a Phase-24 decision.
- **`AskGlobal` vs an `Ask` mode flag.** This summary recommends a distinct `AskGlobal` method. If the roadmapper prefers one entry point, a `GlobalMode bool` on `AskOptions` is acceptable as long as the global path still bypasses retrieve/rerank/pack internally.
- **Report cache lifetime.** In-memory cache (per-`System`, lost on restart) vs persisted in `postgres` `_community_reports`. Recommend: persist when a `GraphStore`/`CommunityStore` is present, in-memory otherwise — mirroring how the graph itself degrades.
- **Resolution ordering with provenance.** The fuzzy pre-pass rewrites names before `Canonicalize`; confirm `SourceChunkIDs` provenance still unions correctly through the rewrite (it should — `Canonicalize` already unions provenance on merge).
