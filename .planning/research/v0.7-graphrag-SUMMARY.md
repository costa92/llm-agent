# Research Summary: v0.7 GraphRAG — Relationship-Traversal Retrieval

**Target:** `llm-agent-rag` (`github.com/costa92/llm-agent-rag`), standalone Go RAG SDK at tag `v0.3.0`
**Milestone direction:** GraphRAG — extract a knowledge graph (entities + relations) from ingested documents and retrieve by traversing it.
**Researched:** 2026-05-18
**Overall confidence:** HIGH on architecture/seam fit (grounded in codebase scan); HIGH on the GraphRAG spectrum and storage tradeoffs (current sources); MEDIUM on cost/quality numbers (vendor blogs, directionally consistent).

---

## Executive Summary

GraphRAG is a family, not a single design. It ranges from **lightweight** (LLM extracts entities per chunk, chunks are linked through shared entities, retrieval expands a vector hit's entity neighborhood) to **full Microsoft-GraphRAG** (LLM entity/relation extraction → typed graph → Leiden hierarchical community detection → LLM-generated community summaries → map-reduce global search + entity-anchored local search). The full pipeline is powerful for whole-corpus "sense-making" questions but is expensive (entity extraction alone is ~58% of indexing tokens per current reporting), slow to update, and hard to evaluate. The community consensus in 2026 is to **start vector/hybrid + structured metadata, then layer graph retrieval onto the specific queries that demonstrably need cross-entity traversal** — exactly the position `llm-agent-rag` is in after v0.6.

`llm-agent-rag` is unusually well-prepared for a *bounded* GraphRAG milestone. The v0.6 architecture already gives the three seams GraphRAG needs to slot into: a `retrieve.Retriever` interface (`Retrieve(ctx, Request) ([]store.Hit, Trace, error)`) that decorators compose over (`HybridRetriever`, `MultiHopRetriever`, `VariantRetriever`); a `store.Store` seam with an **optional-capability** extension pattern (`store.LexicalSearcher`, type-asserted with an in-process fallback); and `generate.Model` / `embed.Embedder` LLM seams that are always unit-tested against a scripted model. GraphRAG should follow these grooves verbatim: a new `EntityExtractor` seam over `generate.Model`, a new `GraphStore` *optional capability* mirroring `LexicalSearcher`, and a new `GraphRetriever` that fuses into the existing RRF pipeline as a fourth signal.

**The keystone decision is storage.** The honest recommendation: **do NOT add a graph database. Build the graph as (a) a stdlib in-memory adjacency structure for the default store, and (b) two relations tables + recursive CTE traversal inside the existing `postgres` subpackage.** The repo's measured constraint — "mostly stdlib, the only non-stdlib island is `postgres`, new deps must follow the `postgres` pattern" — combined with current evidence that recursive CTEs handle multi-hop traversal in microseconds below ~50K edges, means a Neo4j subpackage would add a heavyweight operational dependency to solve a scale problem this SDK does not have. Neo4j stays a *documented future seam point*, not v0.7 scope.

The minimum coherent slice is **lightweight GraphRAG**: deterministic-and-LLM entity extraction at ingest time, a graph store with 1–2 hop neighborhood expansion, and a `GraphRetriever` that fuses as a graph signal. Microsoft-style community detection and global search are explicitly **out of scope** for v0.7 — they are a believable v0.8.

## 2. The GraphRAG Spectrum — and where v0.7 should sit

| Tier | Construction | Retrieval | Cost / update | Fit for a *first* milestone |
|---|---|---|---|---|
| **0. Entity-tagged chunks** | NER tags entities onto chunk metadata | metadata filter by entity | trivial | Too thin — already expressible with v0.6 metadata filters. |
| **1. Lightweight GraphRAG** (LightRAG-ish) | LLM extracts entities + relations per chunk; entities link chunks; incremental union-merge on re-ingest | query → link to entities → 1–2 hop neighborhood expansion → collect chunks of neighbor entities; fuse with dense/lexical | extraction is one LLM call per chunk at ingest; updates are cheap (union new subgraph) | **← RECOMMENDED v0.7 target.** Minimum coherent + genuinely useful slice. |
| **2. Path/subgraph retrieval** | tier 1 + typed relations + relation weights | multi-hop path ranking, subgraph extraction, path-as-evidence | extraction cost same; traversal cost grows | Partial — typed relations yes; deep path ranking is v0.8. |
| **3. Full Microsoft GraphRAG** | tier 1 + Leiden hierarchical community detection + LLM community summaries | global search (map-reduce over community reports) + local search (entity-anchored) + DRIFT | community summarization is a second large LLM pass; entity extraction ≈58% of indexing tokens; slow refresh | **Out of scope for v0.7.** Believable v0.8. |

**Where v0.7 sits: Tier 1, reaching into the typed-relations part of Tier 2.** This is the LightRAG end of the spectrum. Rationale:

1. **It is the minimum coherent slice.** Tier 0 is already doable with v0.6 metadata filters; it is not a milestone. Tier 1 is the first point where the *graph itself* does retrieval work — entity linking + neighborhood expansion surfaces chunks that vector/lexical similarity misses (multi-hop facts, "who else worked with X", entity bridges).
2. **It respects the storage constraint.** Tier 1's graph is small and local; it fits an in-memory adjacency map and a 2-table SQL schema. Tier 3's community detection wants a real graph engine and a heavy second LLM pass.
3. **It is evaluable.** Tier 1 graph hits fuse into the existing RRF + RAG-Triad eval harness — you can measure whether the graph signal improves recall. Tier 3 global search needs new whole-corpus eval methodology.
4. **Current consensus backs incrementalism.** Multiple 2026 production write-ups: "start with vector RAG plus structured metadata, then layer GraphRAG onto the specific use cases that demonstrably need cross-entity traversal." Microsoft itself shipped LazyGraphRAG (defer community summarization to query time, ~0.1% of standard indexing cost) — a signal that even the originators treat full GraphRAG indexing cost as a problem to design around.

**Explicitly deferred to v0.8:** Leiden community detection, LLM community summaries, map-reduce global search, DRIFT search.

## 3. Knowledge-Graph Construction

**Extraction approach.** One LLM call per chunk (or per small chunk-batch) over `generate.Model`, mirroring `advanced.ExpandQuery`/`GenerateHypothetical`: a fixed system prompt asks for entities (`name`, `type`, short `description`) and relations (`source`, `target`, `relation`, optional `description`). Two parse strategies, both already precedented in the repo:
- **Line-delimited** (like `advanced` helpers / `parseSubQueries`) — simplest, robust.
- **Lenient JSON** (like `eval.LLMJudge` JSON parsing) — richer, recommended; tolerate fences/preamble.

A `tuple-delimited` format (LightRAG/MS-GraphRAG style: `("entity"<|>NAME<|>TYPE<|>DESC)`) is a good middle ground — easy to parse, compact, one record per line.

**Deterministic-test strategy (non-negotiable per repo discipline).** `EntityExtractor` is LLM-backed, so:
- Unit-test `LLMEntityExtractor` against a scripted model returning canned extraction text; assert the parser produces the expected entities/relations. Cover malformed output (missing fields, junk lines, duplicate entities) — the parser must be lenient and deterministic.
- Provide a **deterministic non-LLM extractor** for tests and zero-LLM users — e.g. `RegexEntityExtractor` or `DictionaryEntityExtractor` (capitalized-noun heuristic or a caller-supplied gazetteer). Mirrors `HeuristicDecomposer` vs `LLMDecomposer` in `multihop.go`. This also gives the in-memory default path a no-LLM story.

**Entity resolution / canonicalization.** Same surface form across chunks must merge to one node. v0.7 scope: case-fold + trim normalization + exact-match merge on `(normalized name, type)`, with descriptions concatenated/deduped (MS-GraphRAG merges descriptions; LightRAG unions). Embedding-similarity fuzzy resolution ("Acme" ≈ "Acme Corp") is a **known limitation, deferred** — call it out explicitly.

**Incremental updates on re-ingest.** This is a first-class requirement, not a follow-up — re-ingest already exists (`ImportOptions.ReplaceSource`, `RemoveByFilter`). Design:
- Each entity/relation records its source chunk IDs (provenance).
- Re-ingesting a document: remove that document's graph contributions, re-extract, union the new subgraph in. An entity referenced only by removed chunks is garbage-collected; an entity still referenced by other chunks survives with its other provenance intact.
- This is LightRAG's "union new documents into the existing graph" model and keeps updates cheap.

## 4. Graph-Traversal Retrieval

The `GraphRetriever` (a `retrieve.Retriever`) pipeline:

1. **Query-side entity linking.** Map the query to seed entities in the graph. Two strategies, both in-repo precedent:
   - lexical/normalized-name match (no embedder needed — keeps in-memory path zero-LLM);
   - embed query + cosine vs entity-name/description embeddings (better recall; reuses `embed.Embedder`).
   Provide both behind a small `EntityLinker` seam; default to lexical.
2. **Neighborhood expansion.** From seed entities, expand 1–2 hops over relations (depth configurable, **default 1, hard cap 2 in v0.7**). Collect neighbor entities and the chunks they came from.
3. **Subgraph → chunk hits.** Convert reached entities into their provenance chunks, scored by graph proximity: seed entities highest, decaying per hop (`score = base * decay^hop`). Optionally weight by relation count / edge strength. This is structurally the same shape as `expandSectionLeaves` depth-decay scoring in `retrieve.go`.
4. **Fusion.** `GraphRetriever` returns `[]store.Hit` + a graph `Trace`. It fuses into `HybridRetriever` as a **fourth RRF signal** alongside dense/lexical/structure — same `1/(k+rank)` formula, plus a `GraphRank` field on `FusionAttribution`. RRF is signal-count-agnostic, so this is a clean extension.

**Path ranking / subgraph-as-evidence:** v0.7 keeps it simple — proximity-decay scoring only. Ranking *paths* (not just nodes) and emitting subgraphs as structured evidence is Tier 2 depth → **deferred**.

**Trace.** Extend `retrieve.Trace` with a graph attribution block: linked seed entities, traversed entity IDs, hop depth reached, and per-hit graph provenance. Mirrors how `Fusion` and `Hops` already attribute retrieval. Surface through `rag.Diagnostics`.

## 5. STORAGE — The Keystone Decision

**Question:** how to store and traverse the graph. Three options weighed against the mostly-stdlib constraint and the `postgres`-subpackage precedent.

### Option A — Graph layer over `store.Store` / relations table in `postgres` (recursive CTEs)
- In `postgres`: two new tables (`entities`, `relations`), created in the existing idempotent `Migrate()`. Multi-hop traversal via **`WITH RECURSIVE`** CTEs with a depth bound.
- **Pros:** zero new dependencies (pgx already present); reuses the `postgres` island exactly; transactional consistency with chunks; current evidence — recursive CTEs traverse 2–3 hops in microseconds below ~50K edges, which covers this SDK's target scale comfortably; entities/chunks co-located so provenance joins are trivial.
- **Cons:** SQL recursive CTEs are less expressive than Cypher for constrained path patterns; deep traversal on very large graphs is slower than native graph storage (not a v0.7 concern given depth cap 2).

### Option B — In-memory graph for the default store
- Pure-stdlib adjacency structure: `map[entityID]*Entity`, `map[entityID][]Relation`. Hops are plain map walks.
- **Pros:** zero dependencies; trivially deterministic and unit-testable; fast at SDK-default scale; satisfies the "in-memory default stays pure stdlib" rule.
- **Cons:** not persistent (acceptable — the in-memory `store.Store` is already non-persistent); rebuilt from ingest each run.

### Option C — Dedicated graph database (Neo4j etc.) as a new isolated subpackage
- **Pros:** purpose-built traversal (index-free adjacency), Cypher expressiveness, scales to deep multi-hop on huge graphs.
- **Cons:** a **heavyweight new operational dependency** — even isolated behind a subpackage/build tag, it changes the SDK's deployment story (run a graph DB). Current sources: graph DBs "struggle with real-time queries on large or frequently updated datasets" and a hybrid (dense for speed, graph for refinement) is usually the practical answer. Solves a scale problem `llm-agent-rag` does not yet have. The `postgres` precedent exists because pgvector is *necessary* for production vector search at scale; a graph DB is not similarly necessary for Tier-1 GraphRAG.

### RECOMMENDATION: **A + B together. Not C.**

Build the graph as a **`GraphStore` optional capability**, exactly mirroring `store.LexicalSearcher`:
- **In-memory default** implements `GraphStore` with a stdlib adjacency graph (Option B).
- **`postgres.Store`** implements `GraphStore` with `entities`/`relations` tables + recursive-CTE traversal (Option A), created in the existing `Migrate()`.
- `GraphRetriever` type-asserts for `GraphStore`. A store that does not implement it → graph retrieval is a no-op / the retriever degrades to its base — same graceful-degradation contract as `LexicalSearcher`'s in-process fallback.

This keeps the in-memory path pure stdlib, adds **zero new dependencies**, reuses the `postgres` island, and matches the v0.6 optional-capability pattern bit-for-bit. **Neo4j is a documented future seam point** — because `GraphStore` is an interface, a `neo4jgraph` subpackage can be added later in full isolation without touching any v0.7 code. Defer it; do not pre-build it.

## 6. Proposed Seams (mirroring how v0.6 introduced seams)

New package **`graph`** (peer of `retrieve`, `rerank`, `eval`):

```go
package graph

// Entity is a node. Type is a free-form label ("person", "org", ...).
type Entity struct {
    ID, Name, Type, Description string
    SourceChunkIDs []string          // provenance — drives GC on re-ingest
    Metadata map[string]any
}

// Relation is a directed, typed edge.
type Relation struct {
    ID, Source, Target, Relation, Description string
    SourceChunkIDs []string
    Weight float64
}

// EntityExtractor turns a chunk's text into graph primitives.
// LLM-backed impls are unit-tested with a scripted generate.Model.
type EntityExtractor interface {
    Extract(ctx context.Context, chunkID, text string) ([]Entity, []Relation, error)
}
// LLMEntityExtractor{Model generate.Model}     — prompt + lenient parse
// DictionaryEntityExtractor / RegexExtractor   — deterministic, zero-LLM

// EntityLinker maps a query to seed entities in the graph.
type EntityLinker interface {
    Link(ctx context.Context, query string, g GraphStore) ([]Entity, error)
}
// LexicalEntityLinker (default, no embedder) ; EmbeddingEntityLinker
```

`GraphStore` — **optional capability interface**, lives in `store` next to `LexicalSearcher`:

```go
package store

// GraphStore is an optional capability a Store may implement to persist and
// traverse an entity/relation graph. Graph retrieval type-asserts for it and
// degrades gracefully when a store does not implement it.
type GraphStore interface {
    UpsertGraph(ctx context.Context, ns string, ents []graph.Entity, rels []graph.Relation) error
    RemoveGraphBySource(ctx context.Context, ns string, chunkIDs []string) error  // incremental re-ingest
    Neighborhood(ctx context.Context, ns string, seedIDs []string, depth int) (Subgraph, error)
    FindEntities(ctx context.Context, ns string, names []string) ([]graph.Entity, error)
}
```
(If a `store→graph` import cycle is awkward, the graph types live in `graph` and `store` imports `graph`; `graph` must not import `store`. Settle this in the phase plan.)

`GraphRetriever` — a `retrieve.Retriever` in the `retrieve` package:

```go
type GraphRetriever struct {
    Linker    graph.EntityLinker
    Store     store.Store     // type-asserted for store.GraphStore
    MaxDepth  int             // default 1, hard cap 2
    HopDecay  float64         // proximity score decay per hop
}
func (r GraphRetriever) Retrieve(ctx, req) ([]store.Hit, Trace, error)
```

Fuses into `HybridRetriever` as a fourth signal; `FusionAttribution` gains `GraphRank int`.

Additive-only changes elsewhere: `ImportResult` gains graph counts; `retrieve.Trace` gains a graph attribution block; `rag.Diagnostics` surfaces it. **No existing seam changes shape.**

## 7. Production Pitfalls & How v0.7 Pre-empts Them

| Pitfall (from current sources) | How it bites | v0.7 mitigation |
|---|---|---|
| **Extraction cost** — LLM entity extraction is ~58% of indexing tokens; scales with doc count *and* entity density | Ingest becomes slow/expensive | One call per chunk (batchable); ship a deterministic non-LLM extractor as the zero-cost default; instrument token counts in `ImportResult` (v0.6 `obs` already does cost metrics — reuse). |
| **Graph bloat** — every lookup value becomes a node; graph explodes | Huge graph, slow traversal | Extraction prompt guidance: extract *salient* entities, not every noun. Document the "if it's a lookup value, it's a property not a node" principle. Cap entities-per-chunk. |
| **Traversal explosion** — one hub entity fans out to thousands at each hop; uncapped 5-hop queries blow past 10s | Latency spikes, runaway queries | **Hard depth cap 2 in v0.7** (default 1). Cap neighbors-expanded-per-hop (fan-out limit). Bound the recursive CTE with explicit depth + row limits. |
| **Entity-resolution drift** — same entity, many surface forms; or schema drift over time | Graph fragments; recall silently degrades | v0.7: deterministic case-fold/exact-merge canonicalization + provenance. Fuzzy embedding resolution explicitly deferred and documented as a known limitation. |
| **Eval difficulty** — graph retrieval quality is hard to measure | Can't tell if the graph helps | Fuse graph as an RRF signal so it runs through the existing RAG-Triad harness; add a graph-recall metric to `eval`; A/B graph-on vs graph-off on the eval dataset. |
| **Incremental-update rot** — graphs without refresh drift 15–20%/quarter from ground truth | Stale graph | `RemoveGraphBySource` + union-merge on re-ingest from day one (Keystone 5). Provenance-based GC. |
| **Over-adoption** — GraphRAG everywhere when 80% of queries are fine with vector search | Wasted cost/latency | Graph is an *opt-in fused signal*, never the only retriever. Default `rag.System` behavior unchanged unless graph is explicitly wired. |
| **Determinism regressions** — LLM extraction is non-deterministic | Flaky tests | Scripted-model unit tests for the parser; deterministic extractor for everything else (repo discipline). |

## 8. Proposed Milestone Shape (v0.7)

### Scope

**IN:**
- `graph` package: `Entity`, `Relation`, `EntityExtractor` seam, `EntityLinker` seam.
- LLM entity/relation extraction over `generate.Model` (`LLMEntityExtractor`) + a deterministic non-LLM extractor.
- `store.GraphStore` optional-capability interface; in-memory (stdlib) impl; `postgres` impl (recursive-CTE traversal, tables in `Migrate()`).
- Deterministic exact-match entity canonicalization + provenance.
- Incremental graph update on re-ingest (`RemoveGraphBySource` + union-merge).
- `retrieve.GraphRetriever` (entity linking → 1–2 hop expansion → proximity-decay scoring).
- Graph as a fourth RRF signal in `HybridRetriever`; `GraphRank` attribution.
- Graph attribution in `retrieve.Trace` + `rag.Diagnostics`; graph counts in `ImportResult`.
- Graph-recall metric in `eval`; graph-on/off A/B in the Triad harness.
- A worked example using the scripted model (deterministic, per repo convention).

**OUT (explicitly deferred to v0.8+):**
- Leiden / hierarchical community detection.
- LLM-generated community summaries.
- Map-reduce **global search** and **DRIFT** search.
- Embedding-similarity **fuzzy entity resolution** / coreference.
- Neo4j or any dedicated graph-DB subpackage (kept as a documented future `GraphStore` impl).
- Path-ranking / structured subgraph-as-evidence output (only node-proximity scoring in v0.7).
- Claim/covariate extraction (MS-GraphRAG's third primitive).

### Keystone Decisions (decision points for the roadmapper to ratify)

- **KG-1 — Spectrum position.** v0.7 targets **Tier 1 lightweight GraphRAG** (entity/relation extraction + neighborhood-expansion retrieval), reaching into typed relations. Community detection / global search are v0.8.
- **KG-2 — Storage.** Graph is a **`GraphStore` optional capability** (mirroring `store.LexicalSearcher`): stdlib in-memory impl + `postgres` recursive-CTE impl. **No graph database in v0.7.** `GraphStore` being an interface keeps Neo4j a clean future seam point.
- **KG-3 — Extraction is dual-mode.** An `EntityExtractor` seam with an LLM impl (`generate.Model`, scripted-model tested) **and** a deterministic non-LLM impl — mirroring `LLMDecomposer`/`HeuristicDecomposer`. The default path stays zero-LLM-capable.
- **KG-4 — Fusion, not replacement.** `GraphRetriever` slots in as a **fourth RRF signal** under `HybridRetriever`; it never replaces dense/lexical retrieval. Default `rag.System` behavior is unchanged unless graph is explicitly wired.
- **KG-5 — Incremental updates from day one.** Re-ingest must reconcile the graph (provenance-based removal + union-merge), not just append. This is in v0.7 scope, not a follow-up.
- **KG-6 — Canonicalization is deterministic-only in v0.7.** Exact-match + case-fold merge on `(name, type)`. Fuzzy/embedding entity resolution is a documented known limitation deferred to v0.8.
- **KG-7 — Traversal is hard-bounded.** Depth cap 2 (default 1) and a per-hop fan-out cap, enforced in both the in-memory walk and the recursive CTE. No uncapped traversal ships.

### Candidate Phase Breakdown (~2–3 slices/phase, matching v0.6 cadence)

**Phase A — Knowledge-graph construction**
- `A-01` `graph` package + `EntityExtractor` seam; `LLMEntityExtractor` over `generate.Model` (prompt + lenient parse), scripted-model unit tests covering malformed output.
- `A-02` Deterministic `DictionaryEntityExtractor`/`RegexEntityExtractor`; exact-match canonicalization + provenance; extraction wired as a post-split ingest stage with counts in `ImportResult`.

**Phase B — Graph storage (the keystone)**
- `B-01` `store.GraphStore` optional-capability interface + stdlib in-memory implementation (adjacency graph, `Neighborhood` walk with depth/fan-out caps).
- `B-02` `postgres` `GraphStore`: `entities`/`relations` tables in `Migrate()`, recursive-CTE `Neighborhood` traversal; graph conformance suite shared across both impls (mirrors the v0.6 lexical conformance suite).
- `B-03` Incremental update: `RemoveGraphBySource` + union-merge on re-ingest, provenance GC; tested across both stores.

**Phase C — Graph-traversal retrieval & fusion**
- `C-01` `EntityLinker` seam (`LexicalEntityLinker` default, `EmbeddingEntityLinker`); `retrieve.GraphRetriever` with proximity-decay scoring.
- `C-02` Fuse graph as a fourth RRF signal in `HybridRetriever`; `GraphRank` in `FusionAttribution`; graph attribution in `Trace` + `rag.Diagnostics`.
- `C-03` Graph-recall metric in `eval`; graph-on/off A/B in the Triad harness; a deterministic scripted-model example + docs (including the "Neo4j is a future `GraphStore`" note and the v0.8 deferral list).

## Gaps / Open Questions for the Roadmapper

- **`store`↔`graph` import direction.** `GraphStore` referencing `graph.Entity` may force `store` to import `graph`. Confirm `graph` never imports `store` to avoid a cycle; resolve in the Phase B plan.
- **Batching.** Whether `EntityExtractor.Extract` is per-chunk or accepts a chunk batch (cost vs. context-window tradeoff) — a Phase A plan-time decision.
- **Namespace semantics.** Whether the graph is per-namespace (recommended, matches `store` namespacing) or global — assume per-namespace unless contradicted.
- **`MultiHopRetriever` vs `GraphRetriever` naming.** Both are "multi-hop." Keep names distinct (query-decomposition hops vs graph hops) to avoid user confusion; flag in docs.
