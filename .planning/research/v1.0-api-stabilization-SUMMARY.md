# Research Summary: v1.0 — API Stabilization & the Compatibility Promise

**Target:** `llm-agent-rag` (`github.com/costa92/llm-agent-rag`), standalone Go RAG SDK at tag `v0.6.0`
**Milestone direction:** Lock the public API and make a Go-module compatibility promise. A **stabilization / quality milestone — no major new features.**
**Researched:** 2026-05-19
**Overall confidence:** HIGH on the codebase audit (grounded in the `v0.6.0` source); HIGH on the scope recommendation (`llm-agent-rag` v1.0.0 only); HIGH on the Go-compatibility-promise mechanics; MEDIUM on the exact slice split inside each phase.

---

## Executive Summary

`llm-agent-rag` has grown from `v0.1.0` to `v0.6.0` across six milestones (v0.5→v0.9 in this repo's planning numbering). It now spans the full practical GraphRAG spectrum: import/split/embed/store, hybrid + structure-aware + graph retrieval, RRF fusion, reranking, packing, generation, the RAG-Triad eval harness, lightweight + Tier-3 GraphRAG (communities, global search, DRIFT), and a postgres/pgvector backend. The SDK works and is well-tested. What it has **not** done is *stop changing its public surface* — every milestone v0.5→v0.9 added exported types. v1.0 is the milestone where that stops: the API is audited, the accidental and inconsistent parts are fixed, and the module commits to the Go import-compatibility promise.

**The headline finding is scope discipline of a different kind than v0.7–v0.9.** Those were feature milestones and the discipline was "ship two of three, defer the third." v1.0's discipline is "**touch nothing that does not need touching, and freeze the rest.**" The temptation in a stabilization milestone is to keep polishing — rename for elegance, refactor for symmetry, add the one missing convenience. v1.0 must resist that: every API change made *now* is free, every API change made *after* v1.0 is a breaking change requiring `v2`. So v1.0 is a one-time, deliberate, bounded audit-and-fix pass, then a freeze.

**The recommended scope is `llm-agent-rag` v1.0.0 only** (plus a light compatibility-policy doc in the core `llm-agent/rag` facade). `llm-agent-rag` is the active, mature SDK at `v0.6.0`; it has earned a v1.0. The core `llm-agent` module and the sister repos (`llm-agent-otel`, `llm-agent-providers`, `llm-agent-customer-support`) are at different maturity levels and on their own version tracks — bundling them into one "v1.0" is scope creep that dilutes the promise. The `contract/` package already pins the *exact* cross-repo surface the core facade consumes; v1.0's job there is to confirm that surface is part of the frozen API, not to re-version the core.

**The audit found a healthy codebase with a small, fixable set of pre-freeze issues.** No `TODO`/`FIXME`/`XXX`/`HACK`/`Deprecated` markers anywhere in non-test code — there is no dead-code or half-built backlog to clean. No `replace` directives, a `release-precheck` CI gate already rejects them. The cross-repo contract gate already exists. The real issues are: (1) the **`package ragkit` / module-name mismatch** in `doc.go` — the module is `llm-agent-rag` but the root package declares `ragkit`; (2) **11 of 19 packages have no package doc comment**; (3) the `eval` package has grown **four near-parallel evaluator types** (`Evaluator`, `GlobalEvaluator`, `DriftEvaluator`, `TriadEvaluator`) with four parallel result types — a naming-consistency review is warranted; (4) there is **no API-snapshot / apidiff gate** to catch post-v1.0 breakage. None of these are deep; all are exactly the kind of thing v1.0 exists to fix *before* the freeze.

The hard constraints hold trivially: stabilization needs **no new dependency**, holds the stdlib-only line, and keeps `postgres` (and the build-tagged `adapter/llmagent`) as the only non-stdlib islands. v1.0 is, by design, the lowest-dependency-risk milestone in the SDK's history.

## 2. Specific Audit Findings — the three judgement calls

### 2.1 The `ragkit` root package — rename, or delete?

`doc.go` declares `package ragkit` and is the *only* file in the module root. It has a package comment but **no exported symbols** — it is a pure documentation package. The module path is `github.com/costa92/llm-agent-rag`, so Go's default import identifier would be `llm_agent_rag` were the package not explicitly named; it *is* explicitly named `ragkit`.

Two clean options, one must be chosen and frozen:

- **(Recommended) Keep the root as a doc-only package, rename it to match the import path expectation.** Go convention: the root package of `…/llm-agent-rag` is usually named after the last path segment with hyphens dropped — but `llmagentrag` is ugly. The honest fix is: the root package is doc-only and **no one should import it**; rename it to `ragkit` *consistently* by also documenting in the package comment that `ragkit` is the SDK's short name and the root is a doc anchor only. `ragkit` is a fine short brand name. The mismatch is only a problem if it is *undocumented* — so document it.
- **(Alternative) Delete `doc.go` entirely** and let the root be non-importable. Cleaner, but loses the module-overview doc anchor that `pkg.go.dev` shows at the top of the package list.

**Recommendation (KS-3):** keep `doc.go`, keep the name `ragkit`, but make the package comment explicitly state "this root package is a documentation anchor; import the sub-packages." That converts an accidental-looking mismatch into a deliberate, documented decision. It is a one-file change and it is free now / impossible later.

### 2.2 The `eval` package — four evaluators

`eval` grew an evaluator per answer path: `Evaluator` (retrieval recall/MRR/precision — v0.5-era), `GlobalEvaluator` (v0.8 global search), `DriftEvaluator` (v0.9 DRIFT), `TriadEvaluator` (RAG-Triad generation eval). Each has its own result type. The base `Evaluator`/`Result` has no prefix; the three later ones are prefixed.

This is not *broken* — it works and is well-tested — but it is the clearest naming-consistency wart in the SDK, and `eval` is a public package. Three dispositions, Phase-28 must pick one:

1. **Rename the base** `Evaluator`→`RetrievalEvaluator`, `Result`→`RetrievalResult`, `Run` unchanged. Full symmetry. Breaking rename — must happen in v1.0.
2. **Accept the asymmetry**, document it ("`Evaluator` is the retrieval evaluator; the answer-path evaluators are prefixed because they are newer and narrower"). Zero code change.
3. **Unify behind one interface** — a single `Evaluator` interface with implementations. This is a *refactor*, not a rename, and risks scope creep into "v1.0 redesigns eval" — **reject it**; v1.0 freezes, it does not redesign.

**Recommendation (KS-4):** option 1 — rename the base to `RetrievalEvaluator`/`RetrievalResult`. It is a small, mechanical, breaking-but-free-now change that makes the public `eval` surface fully consistent. Reject option 3 as out of scope.

### 2.3 Accidental exports — sweep result

A scan for symbols that *look* accidentally exported (exported but only used internally, or exported for test convenience). The audit found the surface **largely intentional**: the many small seam interfaces (`EntityLinker`, `QueryDecomposer`, `SectionPlanner`, `PathRanker`, etc.) are deliberate plug-points, not accidents. Two items warrant a Phase-28 ruling:

- `pack.TokenCounter` / `pack.SimpleCounter` — exported, and `SimpleCounter{}` is used by `rag/ask.go`. It *is* a legitimate seam (a caller can supply a real tokenizer). Keep, but confirm.
- `retrieve` exposes 6+ concrete retriever structs and 5+ seam interfaces. All are intentional. The Phase-28 audit should *record* "this is the frozen retrieve surface" rather than trim it.

There is no obvious accidental export — but the Phase-28 audit must produce a **written, reviewed inventory** so "we looked at every exported symbol and decided to keep it" is a recorded fact, not an assumption.

## 3. The Go Compatibility Promise — what v1.0 commits to

`llm-agent-rag` is a Go module. Tagging `v1.0.0` invokes the **Go 1 compatibility promise** *as applied to this module's own API*, governed by three rules every contributor and user must understand. v1.0's deliverable here is a `docs/compatibility.md` (or `COMPATIBILITY.md`) that states all three.

1. **The import-compatibility rule (the core of it).** *"If an old package and a new package have the same import path, the new package must be backward-compatible with the old package."* Concretely, after `v1.0.0`, within the `v1.x.y` series, no PR may: remove or rename an exported symbol; change an exported function/method signature; remove a field from an exported struct; add a method to an exported *interface* (that breaks every implementer); change exported constant values that callers depend on. **Additive-only** within v1: new functions, new struct *fields* (callers using keyed literals are unaffected), new packages.

2. **Semver, as Go enforces it.** `v1.MINOR.PATCH`: PATCH = bug fixes, MINOR = backward-compatible additions, MAJOR = breaking changes. This is not advisory — Go's module system *enforces* it via rule 3.

3. **The major-version-suffix rule.** A `v2+` module **must** change its import path: `github.com/costa92/llm-agent-rag/v2`. There is no other way to ship a breaking change. This is the safety net: a breaking change is not "discouraged," it is *impossible* without a visible, opt-in path change. So the cost of getting v1.0 wrong is real — a mistake means either living with it through all of v1, or a disruptive `/v2`.

**What the policy doc should say (KS-5):**
- The exact additive-only rules above, with the interface-method gotcha called out explicitly.
- That the **`contract` package's pinned surface is a hard sub-contract** — it changes only via coordinated PRs with `llm-agent`.
- The `postgres` package's external deps (`pgx`, `pgvector-go`) — pin a *minimum* version policy; their major bumps are *their* semver, surfaced through `go.mod`.
- A statement that **the build-tagged `adapter/llmagent` is covered by the promise too** — build tags do not exempt a package from semver.
- The deprecation procedure: how something *would* be retired in a future `/v2` (mark with a `// Deprecated:` comment in a v1.x minor, remove only in v2).
- Pre-v1.0, link `docs/core-compatibility.md` (the cross-repo story already exists).

## 4. Documentation Completeness — the v1.0 doc bar

Current state: a 188-line `README.md`, four `docs/` files (`backend-selection.md`, `core-compatibility.md`, `graphrag.md`, `production-deployment.md`), five worked `examples/*_test.go`, and **8/19 packages with doc comments**.

A v1.0 doc bar — what "documented" means for a frozen public SDK:

1. **Every package has a package comment.** (Fix A2 — 11 missing.) The `rag` package comment is the most important: it is the front door and currently blank on `pkg.go.dev`.
2. **Every exported type, function, method, and `Err*` value has a doc comment** starting with the symbol name (Go convention; `gofmt`/`go doc` rely on it). The audit shows `rag/options.go` already does this well for `GlobalOptions`/`DriftOptions`/`Options` fields — that quality bar applied package-wide. A Phase-29 sweep should run `go doc` over every package and fill gaps.
3. **The `README` is corrected for v1.0.** The current README's "Not implemented yet" list still names "cross-repo contract-drift CI gates (planned in slice 13-04)" — but `contract/` *exists*. The "Status: production-ready core, evolving ecosystem" line should become "Status: stable, v1.0" with the compatibility promise linked. The README must not over- or under-claim at the freeze.
4. **`docs/compatibility.md` exists** (§3) — the new v1.0 artifact.
5. **The worked examples compile and run as `go test`** (they already do — `examples/*_test.go`). v1.0 should confirm there is at least one example per *primary* answer path (`Ask`, `AskGlobal`, `AskDrift` — covered; `Retrieve`-only — covered by `basic_import_and_ask_test.go`). Examples are doc *and* a de-facto API-usage snapshot.
6. **`CHANGELOG.md`** gets a `[v1.0.0]` entry framed as "API freeze + compatibility promise," explicitly listing any rename done in Phase 28 (e.g. the `eval` base rename) as the *last* breaking changes.

Documentation is not part of the frozen API surface, so doc gaps are technically fixable post-v1.0 — but shipping a v1.0 with blank package pages is a quality failure. Treat the doc sweep as a release-blocking part of the milestone.

## 5. Cleanup Before Locking — small, because the codebase is clean

The brief asks for deprecations to remove, dead code, `TODO`s, half-built features. **The audit found essentially none of this.** Zero `TODO`/`FIXME`/`XXX`/`HACK`/`Deprecated` markers in non-test code. No `replace` directives. No obviously dead exported symbol.

The only "cleanup" items are documentation-truth and consistency, not code removal:
- The README "Not implemented yet" list is **stale** — `contract/` and the feedback recorder both exist now. Correct it.
- The `eval` naming asymmetry (§2.2) — a rename, not a removal.
- The `ragkit` mismatch (§2.1) — a clarification, not a removal.

**This is a finding, not a gap:** v1.0's "cleanup" phase is light precisely because v0.5–v0.9 were disciplined milestones. The risk in this phase is *inventing* cleanup work (refactors dressed as cleanup). Hold the line: if it is not stale-doc, a naming-consistency fix, or a confirmed-accidental export, it does not belong in v1.0.

## 6. API-Stability Verification — the apidiff gate

The SDK has a **cross-repo** contract gate (`contract/contract_test.go`) — but it pins only the *subset* of symbols the core `llm-agent/rag` facade consumes (roughly: parts of `embed`, `generate`, `ingest`, `prompt`, `rag`, `store`, `retrieve`). It is a compile-time pin and an excellent pattern, but it is **not a full-surface gate** — a PR could rename `graph.WeightedPathRanker` or drop a field from `eval.TriadResult` and the contract test would not notice.

v1.0 needs a **full exported-surface snapshot gate** so that *every* post-v1.0 PR that breaks the API fails CI. Three options:

1. **`golang.org/x/exp/apidiff`** — the canonical tool; compares two versions' exported APIs and classifies changes as compatible/incompatible. **But** it would be a new dependency / tool — and the brief says no new dependency *for the stabilization work itself*. `apidiff` can be run as a `go run` tool without entering `go.mod` as a *library* dep, but it still pulls `golang.org/x/exp` into the *toolchain*. Acceptable as a CI-only `go run`, not as a module require.
2. **A committed API snapshot + a `go test` diff** — generate a deterministic text dump of every exported symbol (via `go/packages` or even a `go doc -all` capture per package), commit it as `api/v1.snapshot.txt`, and a test (or CI step) regenerates and `diff`s. **Pure stdlib** (`go doc` is in the toolchain). This mirrors the `contract` package's philosophy — a committed artifact, a `go test`-time gate, no separate workflow. **Recommended.**
3. **Extend the `contract` package** to pin the *entire* surface, not just the core-facade subset. This is the cheapest conceptually — the pattern already exists — but a hand-maintained compile-pin of ~150+ symbols is tedious and easy to let drift. Better as a *complement* to option 2 than a replacement.

**Recommendation (KS-6):** ship option 2 — a committed `go doc`-based exported-surface snapshot with a `go test` regeneration-diff gate — and keep the existing `contract` package as the narrower cross-repo gate. The snapshot is the *intra*-repo "did this PR break the v1 promise?" gate; `contract` stays the *cross*-repo gate. Both are stdlib, both are `go test`-time, neither needs a new module dependency. The snapshot is generated *once* at the v1.0 freeze and becomes the baseline.

## 7. Scope Decision — what v1.0 covers (and what it does not)

**Recommended scope (KS-1): `llm-agent-rag` v1.0.0 only.**

| Repo | Current state | In v1.0 scope? | Why |
|---|---|---|---|
| `llm-agent-rag` | `v0.6.0`, mature, full GraphRAG SDK | **YES — tag `v1.0.0`** | The active SDK; six milestones of growth; has earned the freeze. |
| `llm-agent` (core) | stdlib-only agents framework, separate version track | **NO — except the compat-policy doc** | Different module, different maturity, its own roadmap. Bundling = scope creep. The `contract` gate already protects the seam. |
| `llm-agent/rag` facade | thin facade consuming the `contract` surface | **Boundary: confirm + document** | v1.0 confirms the `contract`-pinned surface is part of the frozen API and notes the facade dependency in `docs/compatibility.md`. No code re-versioning. |
| `llm-agent-otel` | observer wiring, sister repo | **NO** | Consumes `rag.Observer` — a frozen v1.0 symbol — so it *benefits*, but it is not re-versioned here. |
| `llm-agent-providers` | provider adapters | **NO** | Unrelated to the RAG SDK surface. |
| `llm-agent-customer-support` | reference deployment | **NO** | The deferred deployment layer (HTTP service, CLI) is explicitly out of scope. |

Justification: a "v1.0" that spans five repos at five maturity levels makes a promise no single repo can honor and dilutes what "v1.0" *means*. The clean, honest, bounded scope is: **`llm-agent-rag` freezes its API and tags `v1.0.0`; the core `llm-agent` module continues on its own track; the `contract` package is the documented, gated seam between them.**

**Explicitly OUT of scope for v1.0:**
- **Any new feature.** No new retriever, no new answer path, no new eval metric, no new store backend. v1.0 is a freeze, not a feature milestone.
- **The deferred GraphRAG items** (from v0.9's deferral list): incremental community maintenance, claim/covariate extraction, a dedicated graph database. These stay deferred — to a post-v1.0 `v1.x` (additive ones) or `/v2` (breaking ones).
- **The deployment layer** — HTTP service, CLI. The README's "Not implemented yet" deployment items are *deliberate non-goals*, not v1.0 work.
- **Re-versioning the core `llm-agent` module or any sister repo.**
- **Any `eval` redesign** (§2.2 option 3) — rename only, no refactor.
- **Performance optimization** — v1.0 freezes the API, not the implementation; implementation may still evolve within v1.x.

- **KS-5 — v1.0 ships a written compatibility policy.** A `docs/compatibility.md` stating the Go import-compatibility rule (additive-only within v1, the interface-method gotcha called out), semver, the `/v2`-import-path rule for breaking changes, the `contract`-package sub-contract, the `postgres` external-dep policy, and that the build-tagged `adapter/llmagent` is covered by the promise too.
- **KS-6 — a full exported-surface snapshot gate, stdlib-only.** A committed `go doc`-based snapshot of every exported symbol (`api/v1.snapshot.txt` or similar) with a `go test`-time regeneration-diff gate, generated once at the freeze as the v1 baseline. It complements — does not replace — the existing cross-repo `contract` package. No new module dependency; `apidiff` is rejected as a `go.mod` require (acceptable only as an optional CI-only `go run`).
- **KS-7 — the doc bar is release-blocking.** Every one of the 19 importable packages gets a package comment; every exported symbol gets a name-prefixed doc comment; the stale README "Not implemented yet" list is corrected; `CHANGELOG.md` gets a `[v1.0.0]` entry framing the freeze. A v1.0 with blank `pkg.go.dev` pages does not ship.
- **KS-8 — hard constraints hold trivially.** v1.0 needs no new dependency. stdlib-only holds; `postgres` and the build-tagged `adapter/llmagent` stay the only non-stdlib islands. No `replace` in the `release/v1.0` branch (the `release-precheck` gate already enforces this). `go.sum` stays committed (correct — the module has the postgres-island deps; this differs intentionally from the core `llm-agent`, noted in the policy doc).

## Proposed Phase Breakdown (v1.0 — global numbering continues; v0.9 ended at Phase 27)

**Three phases. v1.0 is a quality milestone, so the phases are an *audit → fix → verify-and-tag* arc rather than the *feature → feature* arc of v0.7–v0.9. Each phase is small — the codebase is clean — but they are sequential: the audit must finish before the fixes, and the fixes before the freeze.**

### Phase 28 — API audit & pre-freeze decisions

**Goal:** produce a written, reviewed inventory of the entire exported surface and *make and record* every naming/consistency decision that must be settled before the freeze. This phase changes little code — it produces decisions and a small set of breaking renames.

- `28-01` Full exported-symbol inventory across all 19 importable packages + `adapter/llmagent` — a written audit doc classifying each symbol *keep / rename / unexport*. Confirms (in writing) that there are no accidental exports.
- `28-02` Apply the ratified breaking renames: `eval.Evaluator`→`RetrievalEvaluator`, `eval.Result`→`RetrievalResult` (KS-4); fix the `ragkit` root package comment (KS-3). Resolve the `Ask*`/`*Options` naming (KS-4 — ratify as-is, recorded). Update all call sites + the `examples/`.
- `28-03` Sweep for stale claims and confirm clean: correct the README "Not implemented yet" list (the `contract` gate and feedback recorder exist); confirm zero `TODO`/`replace`/dead code (already verified — this slice *records* it as a release-readiness fact).

### Phase 29 — Documentation completeness & the compatibility policy

**Goal:** every package and every exported symbol is documented to the v1.0 bar, and the compatibility promise is written down.

- `29-01` Add package doc comments to all 11 missing packages (priority: `rag`, then `store`, `retrieve`, `generate`, `embed`, `ingest`, `pack`, `rerank`, `prompt`, `advanced`, `tree`, `adapter/llmagent`).
- `29-02` Exported-symbol doc-comment sweep — `go doc`-driven gap fill across every package so every exported type/func/method/`Err*` has a name-prefixed comment.
- `29-03` Write `docs/compatibility.md` (KS-5): the import-compatibility rule, semver, the `/v2` rule, the `contract` sub-contract, the `postgres` dep policy, the adapter coverage. Update `README.md` status to "stable, v1.0" and link the policy.

### Phase 30 — API-stability gate, freeze & the v1.0.0 tag

**Goal:** lock the surface with an automated gate, write the changelog, tag `v1.0.0`.

- `30-01` Build the exported-surface snapshot gate (KS-6): a stdlib `go doc`-based generator, the committed `api/v1.snapshot.txt` baseline, and a `go test`-time regeneration-diff. Keep the existing `contract` package as the cross-repo gate; document how the two relate.
- `30-02` Wire the snapshot gate into CI (`test.yml`); confirm `release-precheck.yml` (`replace` rejection) covers the `release/v1.0` branch. Final full-suite verification: `GOWORK=off go vet ./... && go test ./...` + `-tags llmagent`.
- `30-03` `CHANGELOG.md` `[v1.0.0]` entry framing the freeze + listing the Phase-28 renames as the final breaking changes; tag `v1.0.0`.

## Pitfalls & How v1.0 Pre-empts Them

| Pitfall | How it bites | v1.0 mitigation |
|---|---|---|
| **Scope creep — "stabilization" becomes a refactor** | The `eval` four-evaluator review becomes a redesign; "cleanup" invents work | **KS-2:** freeze, not redesign. `eval` gets a *rename* only (KS-4); §2.2 option 3 explicitly rejected. "Cleanup" is scoped to stale-doc + ratified renames. |
| **Bundling sister repos into "v1.0"** | A five-repo promise no single repo can honor; diluted meaning | **KS-1:** scope is `llm-agent-rag` v1.0.0 only; the seam to core is the documented `contract` package. |
| **A breaking change slips in after the freeze** | Forces a disruptive `/v2` or a permanent v1 wart | **KS-6:** the committed exported-surface snapshot + `go test` diff fails any PR that renames/removes/re-signs a symbol. |
| **Adding a method to an exported interface in a "minor" release** | Silently breaks every external implementer — the subtlest semver violation | **KS-5:** the compatibility doc calls this gotcha out explicitly; **KS-6:** the snapshot captures interface method sets, so the diff catches it. |
| **The `ragkit` mismatch frozen as an accident** | `import …/llm-agent-rag` binding `ragkit` is forever surprising and undocumented | **KS-3:** the root package comment is rewritten to *document* `ragkit` as a deliberate doc-anchor name — accident → decision. |
| **Shipping v1.0 with blank `pkg.go.dev` pages** | `rag` and 10 other packages render with no overview — a visible quality failure for a "stable" SDK | **KS-7:** package doc comments are release-blocking; Phase 29 fills all 11. |
| **`apidiff` pulled in as a module dependency** | Violates the no-new-dependency constraint | **KS-6:** the snapshot gate is stdlib `go doc`; `apidiff` is rejected as a `go.mod` require (CI-only `go run` at most). |
| **Stale README claims at the freeze** | A "v1.0" README that says features are unbuilt when they exist (or vice-versa) | `28-03` / `29-03`: the "Not implemented yet" list is corrected; status line updated; no over/under-claiming. |
| **Renaming after the tag** | Any post-`v1.0.0` rename needs `/v2` | **KS-4:** every rename happens in Phase 28, *before* the tag; after `v1.0.0` the surface is frozen. |
| **The `contract` gate mistaken for full coverage** | A PR breaks a non-facade symbol (`graph`, `eval`, …) and nothing catches it | **KS-6:** the snapshot gate covers the *whole* surface; `contract` stays the narrower cross-repo gate — the two are documented as complementary. |
| **`postgres` external deps drift breaking v1 users** | A `pgx` major bump surfaces as a breaking change to v1.0 users | **KS-5:** the compatibility doc states the `postgres` minimum-version policy and that external-dep majors are surfaced via `go.mod`, not absorbed silently. |

## Gaps / Open Questions for the Roadmapper

- **`eval` rename — ratify or reject KS-4?** Rename the base evaluator for full symmetry, or document-and-accept the asymmetry? It is a small breaking change either way; the operator should rule explicitly so Phase 28 has a fixed target.
- **`ragkit` — keep the name or delete `doc.go`?** KS-3 recommends keep-and-document; the operator may prefer deleting the doc-only root entirely. A Phase-28 decision.
- **Snapshot-gate format.** `go doc -all` capture per package vs a `go/packages`-based structured dump. Both are stdlib; `go doc` is simpler, `go/packages` is more precise (captures interface method sets explicitly — relevant to the interface-method pitfall). Recommend `go/packages` if the precision is wanted; a Phase-30 plan-time call.
- **Does the core `llm-agent/rag` facade need a coordinated PR?** If Phase 28 renames any symbol the `contract` package pins, the core facade needs a synchronized update. The audit's recommended renames (`eval.Evaluator`, the `ragkit` comment) do **not** touch the `contract`-pinned set — so likely no coordinated PR is needed, but Phase 28 must confirm this against `contract_test.go` before finalizing renames.
- **`go 1.26.0` minimum.** v1.0 freezes the API; should it also state a minimum Go version policy in `docs/compatibility.md`? Recommend yes — name `go 1.26` as the v1.0 floor and state that minor-Go bumps are allowed within v1.x.
