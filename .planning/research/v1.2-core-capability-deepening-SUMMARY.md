# Research Summary: v1.2 — Core Capability Deepening

**Domain:** Core `llm-agent` (stdlib-only) — new agent-runtime capabilities
**Researched:** 2026-05-20
**Provisional milestone:** v1.2 — "Core Capability Deepening" (theme: **Core v0.6**)
**Core version bump:** `llm-agent v0.5.1 → v0.6.0` (minor — additive only)
**Umbrella version line:** v1.0 → v1.1 → **v1.2**
**Overall confidence:** HIGH for current-state audit (every claim below verified
by direct file read); MEDIUM for the design proposals (the keystone calls fix the
shapes — first-slice prototypes will refine the field-level details)

---

## What this milestone IS

The umbrella has spent six milestones deepening the `llm-agent-rag` SDK
(v0.5→v1.0) and one milestone aligning the ecosystem (v1.1). The **core
`llm-agent` module has been frozen on its v0.4-era agent surface since v0.3
shipped** — every milestone since has either lived in `llm-agent-rag`
(retrieval, GraphRAG, API stabilization) or in sister repos (providers, otel,
refsvc). The core repo's last feature work was v0.3 (Phase 1–7); since then
it has only absorbed a deprecation-removal cycle (v0.4) and a RAG-facade
realignment (v1.1 Phase 31). Validated capabilities — `llm/v2`, the 5 agent
paradigms, `orchestrate`'s 4 multi-agent patterns + `StateGraph`, `memory`'s
3-kind taxonomy, `comm` (MCP/A2A/ANP), `context` (GSSC) — are locked but
**no new agent-runtime primitive has been added in roughly a year of
milestones**.

v1.2 is the first **core-feature** milestone since v0.3. It bumps the core
module to **v0.6.0** (minor — additive). The theme is *deepening*, not
breadth: take 2–3 of the 4 proposed capabilities (multi-agent coordination,
memory tiering, policy/safety middleware, budget/cancellation) to a finished
state, on the existing locked foundation, **stdlib-only**, with no refactor
of validated public types.

This is a **deliberately small milestone** — the surface area is the core
repo only (sister repos stay on their v0.2.x tracks); rag stays a fixed
point (no rag re-tag); and the bias is toward "ship two capabilities
end-to-end" over "stub four". v1.1's discipline holds — alignment-quality,
honest trade-offs, no `replace` directives, no new dependency.

## Current State Audit — what's already in core for each candidate area

This audit is what to **NOT** re-do. Every claim verified against the
working tree at `/home/hellotalk/code/go/src/github.com/costa92/llm-agent-ecosystem/llm-agent`
on 2026-05-20.

### Multi-agent coordination

| Aspect | Finding |
|--------|---------|
| `orchestrate/` patterns | **4 exist:** `Pipeline` (A→B→C), `FanOutFanIn` (planner→workers→aggregator), `RoundRobinChat` (N agents take turns), `RolePlay` (user+assistant 2-agent dialog). Plus `StateGraph[S]` (typed conditional-edge graph, LangGraph-style). |
| `agents.Agent` interface | `Name() / Run(ctx, input) (Result, error) / RunStream(ctx, input) (<-chan StepEvent, error)` (`agent.go:13-21`). Locked public type. |
| `Supervisor` primitive | **Does not exist.** Only `internal/research.Coordinator` (a hand-rolled plan→summarize×N→report pipeline) is named "Coordinator" — *internal-only*. |
| What's missing | A first-class **supervisor↔worker** primitive — a parent agent that **delegates** sub-tasks to workers, observes results, and may re-plan. `FanOutFanIn` is close but it's *plan-then-fan-out-once*; a Supervisor is a *loop* with re-routing. |

**Verdict:** real net-new surface; nothing to demolish. `FanOutFanIn` and
`StateGraph` stay; a `Supervisor` is **additive**.

### Memory tiering

| Aspect | Finding |
|--------|---------|
| `memory/` taxonomy | `Kind` enum has 3 values: `KindWorking`, `KindEpisodic`, `KindSemantic` (`memory.go:23-28`). This is a **cognitive** taxonomy (working = short-term scratchpad; episodic = events; semantic = facts). |
| `Memory` interface | Single interface — `Type / Add / Search / Get / Update / Remove / Stats` (`memory.go:62-69`). |
| `Manager` | Coordinates the 3 cognitive kinds. `Consolidate` promotes from working→episodic by importance. `Forget` is per-kind. |
| Scope/Session/User/Project axis | **Completely orthogonal — does not exist.** No `Scope` field on `MemoryItem`, no per-tenant/per-session/per-user namespace, no propagation through `context.Context`. |

**Verdict:** the proposed "memory tiering (session / project / user — 3 layers)"
is a **different axis** from the existing Working/Episodic/Semantic.
`session/project/user` is *isolation scope*, not *cognitive function*. Calling
it "memory tiering" risks confusing it with the cognitive taxonomy. **Suggest
renaming to "memory scoping"** to keep the dimensions clearly named. The
implementation question (decorator vs interface field vs separate Stores) is
open — see KC-2 below.

### Policy / safety middleware

| Aspect | Finding |
|--------|---------|
| `guard` / `safety` in core | **Does not exist in core.** The only `guard` package in the umbrella lives in **`llm-agent-rag/guard`** (regex PII redaction + injection-pattern scanner, v0.6). It is rag-scoped and operates on retrieved content. |
| Middleware shape elsewhere | `otelmodel.Wrap(inner llm.ChatModel) llm.ChatModel` (Keystone K3) is the canonical *decorator* pattern — capability-preserving (`ToolCaller`/`Embedder`/`StructuredOutputs` assertions survive). |
| Hook pattern in core | **Absent by design.** K3 says "decorator wrappers, never hooks". |
| `OnStep` callback | Per-agent option (`ReActOptions.OnStep`, `FunctionCallOptions.OnStep`, etc.) — observation only, not interception. |

**Verdict:** no core safety primitive exists. A new core-side
**middleware/decorator** is net-new; the existing `otelmodel.Wrap` shape is
the precedent it must compose with. The rag-side `guard` package is **not
moved** into core (it has regex deps — well, `regexp` is stdlib, so
technically OK, but the rag `guard` is scoped to retrieved-content scanning;
core needs a different shape that operates on **agent input / agent output /
tool-call boundary**, not on retrieval).

### Cancellation + budget context

| Aspect | Finding |
|--------|---------|
| `context.Context` cancellation | Used throughout — `ctx.Done` plumbed through `agent.go`, `orchestrate/graph.go`, `orchestrate/roundrobin.go`, `orchestrate/roleplay.go`. Pure stdlib. |
| Step caps (a form of budget) | `MaxSteps` (ReAct, PlanSolve), `MaxParallel` (FunctionCall), `MaxConcurrency` (FanOutFanIn), `MaxTurns` (RoundRobinChat termination), `defaultMaxSteps = 100` (StateGraph) — all **per-agent option fields**, not a shared budget concept. |
| Token budget | Exists **inside `context` package** (`Config.MaxTokens`, `select.go` greedy fill) — for the GSSC prompt-assembly pipeline only. Not propagated to agents or model calls. |
| Wall-clock budget | **Not first-class.** Users must wrap `ctx` with `context.WithTimeout` themselves. |
| Cost/$ budget | **Does not exist.** `agents.Usage` tracks `LLMCalls` + `Tokens` ex post; nothing pre-emptively enforces a spend ceiling. |
| `Result.Usage` | Per-`Run` accumulator (`agent.go:50-55`). Read-only after Run returns; no in-flight enforcement. |

**Verdict:** real net-new surface. **Cancellation is fine**; **budget is
not**. The interesting design question is: do budgets flow through `ctx` as
context-keyed values (cheap, ergonomically Go-idiomatic, but invisible to
the type system), or as a dedicated `Budget` interface (explicit, testable,
but adds a parameter to every agent call)? See KC-4 below.

### Cross-cutting: validated public types in scope of "do not touch"

Per CLAUDE.md hard rule 5/6/7 + PROJECT.md "Validated":

- `llm.ChatModel` interface and `llm.StreamEvent` typed union — **locked** (K1/K2)
- `agents.Agent` interface — locked
- `orchestrate.NodeFunc[S]`, `StateGraph[S]` — locked
- `memory.Memory` interface, `Manager` API — locked (additive extension allowed)
- `otelmodel.Wrap` decorator shape — locked (K3)

Every new surface in v1.2 is **additive** — new packages, new optional
interfaces, new decorators. No edit to the above signatures.

## The 4 candidate capabilities — feasibility per item

### Candidate 1: Multi-agent coordination primitives (Supervisor / Coordinator)

| Aspect | Detail |
|--------|--------|
| Existing surface | `orchestrate.FanOutFanIn` (plan-once, fan out, aggregate); `orchestrate.StateGraph[S]` (typed branching). `internal/research.Coordinator` (hand-rolled, internal). |
| Proposed new surface | `orchestrate/coord` subpackage — or **new top-level functions in `orchestrate/`**: `type Supervisor struct{ planner Agent; workers map[string]Agent; opts SupervisorOptions }`. `Run(ctx, input)` → loop: planner emits a `Dispatch` (worker name + sub-input + done-signal), workers run, supervisor observes `Result`s and re-plans until a planner-emitted `Final` is seen or `MaxRounds` is hit. |
| Stdlib feasibility | **Trivial.** Channels + `context.Context` + a loop. No non-stdlib temptation. |
| Effort | **M.** ~600 LOC (Supervisor + parser for dispatch instructions + per-worker error handling + cancellation propagation + 6-8 tests). Composes with `StateGraph[S]` (Supervisor *is* a constrained StateGraph), so could be implemented as a thin facade over `StateGraph[S]`. |
| Risk | Risk of recreating `FanOutFanIn` with extra round. Mitigation: KC-1 names the distinguishing property — Supervisor is **iterative / re-planning**, FanOut is **one-shot**. |
| Open questions | Where does it live — `orchestrate/` (extends existing pkg) or `agents/coord` (new pkg)? KC-1. Are workers always `agents.Agent`, or could they be other Supervisors? (Suggest: yes — composition.) Should the planner be an LLM or a `func(state) []Dispatch`? (Suggest: both — `PlannerFunc[S]` + an LLM-backed default.) |

### Candidate 2: Memory tiering (session / project / user)

| Aspect | Detail |
|--------|--------|
| Existing surface | `memory.Manager` + 3 cognitive kinds (Working/Episodic/Semantic). `MemoryItem` is flat — no scope/tenant/user field. |
| Proposed new surface | A **`Scope` value** on `MemoryItem` (or carried in `context.Context`) plus a `ScopedStore` decorator that namespaces a wrapped `Memory` by `(Kind, Scope)`. Shape: `memory.Scope{Session, Project, User string}` — empty fields = wildcard. Three-axis lookup: `mem.SearchInScope(ctx, query, topK, scope)`. |
| Stdlib feasibility | **Trivial.** Pure-stdlib namespacing on the underlying `map`. No non-stdlib temptation. |
| Effort | **M.** ~500 LOC if implemented as `ScopedMemory` decorator + a `WithScope(ctx, scope)` helper that propagates through agent calls; ~800 LOC if `Scope` is added as a first-class `MemoryItem` field (touches all 3 kinds + manager + tests). |
| Risk | Nomenclature collision with the cognitive Working/Episodic/Semantic taxonomy. Mitigation: **call it "memory scoping"**, not "tiering" — KC-2. Risk of forcing users into a 3-axis hierarchy they don't need (some apps are user-only, some are session-only). Mitigation: any combination of axes empty = unscoped; default behavior identical to pre-v1.2. |
| Open questions | **Decorator (`ScopedMemory.Wrap(inner)`) vs. interface field (`MemoryItem.Scope`)** — KC-2. How does scope propagate through `agents.Agent.Run`? Suggest: `memory.WithScope(ctx, scope)` adds the scope to `ctx`; the underlying `ScopedMemory` adapter reads it. This avoids adding parameters to every agent call. |

### Candidate 3: Policy / safety middleware

| Aspect | Detail |
|--------|--------|
| Existing surface | None in core. `otelmodel.Wrap` is the decorator-pattern precedent. |
| Proposed new surface | A **`policy` package** in core with: `type RequestGate interface{ Inspect(ctx, req llm.Request) error }` (vetoes bad requests pre-LLM); `type ResponseGate interface{ Inspect(ctx, resp llm.Response) error }` (vetoes bad outputs post-LLM); a `policy.Wrap(inner llm.ChatModel, gates ...Gate) llm.ChatModel` decorator that runs gates and returns sentinel `policy.ErrBlocked`. Audit log via an optional `OnDecision(Decision)` callback. Capability-preserving like `otelmodel.Wrap`. |
| Stdlib feasibility | **Trivial.** Regex (stdlib `regexp`) for the built-in PII / injection patterns ported from rag-guard; everything else is composition. |
| Effort | **M.** ~700 LOC for the package + 2-3 built-in gates (PII redaction, injection detection, max-input-length) + capability-preserving decorator + tests. Streaming case (`Stream` path) is the careful part — gates fire at `EventDone` (output side) but the stream is already partially flushed. KC-3 names how. |
| Risk | The rag-side `guard` already exists; tempting to "just move it". **Don't** — rag-guard is content-scanning at retrieval time; core-policy is request/response interception at the model boundary. Two different cuts, two different packages. The shared regex patterns can be lifted, but the *layer* is different. Risk of reinventing OWASP / NIST safety categories — for v1.2, **scope to the 3 named built-ins**; expand later. |
| Open questions | **Composition with `otelmodel.Wrap`** — `policy.Wrap(otelmodel.Wrap(inner))` vs. `otelmodel.Wrap(policy.Wrap(inner))`? Both must work (capability-preserving wrappers compose). Suggest: docs prescribe outer→inner = policy→otel→provider so denied requests still get observed in spans. Should gates be sync (fail-fast) or async (best-effort logging)? Suggest: sync for `RequestGate`, sync-or-async for `ResponseGate` (denial vs. detection). **KC-3 fixes the interface shape.** |

### Candidate 4: Cancellation + budget context

| Aspect | Detail |
|--------|--------|
| Existing surface | `context.Context` cancellation works; `agents.Usage` is read-only ex post. Per-loop step caps exist scattered as `MaxSteps` / `MaxTurns` / `MaxConcurrency`. |
| Proposed new surface | A **`budget` package** in core: `type Budget struct{ Tokens int; Calls int; Wall time.Duration; Cost float64 }`. Two integration paths under KC-4: **(a) ctx-keyed** — `budget.WithBudget(ctx, b)` + `budget.From(ctx)`; agents call `budget.From(ctx).Charge(usage)` and either return `budget.ErrExhausted` or honor `ctx` cancellation. **(b) explicit `Budget` interface** flowing alongside ctx — `type Tracker interface{ Charge(Usage) error; Remaining() Budget }`, passed as a separate option. |
| Stdlib feasibility | **Trivial — for token/call/wall.** All countable with stdlib. **Cost is harder** — requires per-(provider × model) token-to-dollar conversion; that table is provider-specific and **lives outside core** (the providers sister repo or a user-supplied `CostMapper`). Core ships the *plumbing*; cost-table is opt-in. |
| Effort | **M-L.** ~600 LOC for the budget package + tracker + ctx-keyed helpers + tests. **L** if the integration touches every agent loop (5 paradigms + 4 orchestrators × `MaxSteps`/`MaxTurns`). |
| Risk | Big integration footprint — every agent loop must check `budget.From(ctx)` before each LLM call. Risk of inconsistent enforcement (some agents check, some don't). Mitigation: the budget check goes into `generateFromPrompt` (the shared helper every paradigm uses) — single chokepoint. Streaming case is the careful part — budget enforcement during a stream needs the per-tool-call `Index` to attribute partial usage. Composes with K1 (typed StreamEvent). |
| Open questions | **KC-4 — ctx-keyed vs. dedicated `Budget` interface.** Suggest ctx-keyed for the propagation (Go-idiomatic, zero new function parameters) **plus** an explicit `budget.Tracker` interface for the *enforcement* (so users can swap "strict deny" vs. "soft warn"). Should `MaxSteps` etc. be deprecated in favor of `Budget.Calls`? **No — keep both.** `MaxSteps` is per-agent; `Budget.Calls` is cross-agent (e.g., total LLM calls across a Supervisor + its workers). |

## Recommended milestone scope

### Recommendation: take 3 of the 4 — drop "memory tiering" from v1.2

**v1.2 scope (recommended):**

1. **Supervisor (Candidate 1)** — adds the missing iterative coordination primitive
2. **Policy middleware (Candidate 3)** — adds the missing safety layer at the model boundary
3. **Budget/cancellation (Candidate 4)** — adds the missing in-flight cost/spend governance

**Drop from v1.2:** memory scoping (Candidate 2). Rationale:

- Memory scoping is a **storage-layer** concern; it composes naturally with
  whatever Supervisor/policy/budget land. Adding it now means deciding the
  scope-axis shape (`Session/Project/User`?) before any user has hit
  the pain point — premature.
- The **3 picked items chain naturally:** Supervisor needs cancellation to
  abort a runaway sub-task → that's Budget. Supervisor coordinates multi-step
  workflows → policy gates fire at each step. They form one coherent story
  ("agent runtime governance"). Memory scoping is a separate story.
- Per the GSD-style "deeply finished beats wide-but-shallow" — three
  capabilities in 4 phases is closer to "deeply finished" than four in 5.
- Memory scoping is a strong candidate for **v1.3** — by then there will
  be production telemetry from Supervisor users showing how they actually
  partition memory.

### Ordering within v1.2

1. **First**: Budget/cancellation foundations (Candidate 4 — KC-4). It's the
   *plumbing* that the other two compose with. Supervisor must honor
   budgets across its workers; policy middleware needs a way to charge
   tokens for blocked requests too.
2. **Second**: Policy middleware (Candidate 3 — KC-3). It's the most
   self-contained — a new package, a new decorator, composes with the
   existing `otelmodel.Wrap`. Land it before Supervisor so Supervisor's
   workers can be policy-wrapped from day one.
3. **Third**: Supervisor (Candidate 1 — KC-1). It's the highest-LOC and
   has the most design surface; landing it last means the budget + policy
   primitives are already real (not stubs) and Supervisor's tests can
   exercise them end-to-end.

### Bias toward focus

A 3-capability / 3-phase milestone with deeply finished work is
deliberately smaller than v0.6's six retrieval-quality phases. Two factors:

- **Core surface stays stdlib-only.** Adding tests for each new package
  needs careful work to avoid pulling in any non-stdlib dep (`testify` etc.
  remain banned). The existing `ScriptedLLM` mock + `scriptedllm_test.go`
  pattern works; new tests follow it.
- **No sister-repo work in v1.2.** Sister repos (otel/providers/cs) stay
  on their v0.2.x tracks. The umbrella dep-currency gate (KE-6) will fire
  when sister repos bump core from v0.5.1 → v0.6.0 — that's a *future*
  alignment (v1.3 ecosystem-alignment-style milestone), **not v1.2's job**.

## Keystone Decisions (KC-1 .. KC-5) — for the roadmapper to ratify

The numbering continues the project convention: K* (v0.3, core architecture),
KS-* (v1.0, API stabilization), KE-* (v1.1, ecosystem alignment), **KC-***
(v1.2, **C**ore capabilities).

**KC-1 — Multi-agent coordination lives in `orchestrate/` as `orchestrate.Supervisor`,
not a new `agents/coord` package.** Rationale: it composes with `StateGraph[S]`
and `FanOutFanIn`; users already import `orchestrate` for multi-agent
workflows; a new subpackage fragments the discoverability story. The
Supervisor is implemented as a thin facade over `StateGraph[S]` (the
loop body is a typed state machine — planner-emits-Dispatch → worker-runs
→ planner-observes → repeat-or-finish). This keeps the `StateGraph[S]`
abstraction working and lets Supervisor inherit its cancellation/MaxSteps
guard-rails. Public surface: `func NewSupervisor(opts SupervisorOptions) *Supervisor`,
`(*Supervisor).Run(ctx, input) (Result, error)`. Workers are `agents.Agent`
(so a Supervisor can supervise another Supervisor — composition).

**KC-2 — Memory tiering is OUT of v1.2 scope.** The 4-candidate sweep
recommends deferring it (see "Recommended scope" above). If memory scoping
does land in a later milestone, the **decorator shape** (`memory.ScopedStore(inner Memory, scope Scope) Memory`)
is recommended over interface-field shape (adding `Scope` to `MemoryItem`):
the decorator is purely additive (existing `MemoryItem` callers see nothing),
and `memory.WithScope(ctx, scope)` carries the scope through `ctx`. **This
keystone is a pre-decision; nothing in v1.2 implements it.** Recording it
now so v1.3 doesn't relitigate.

**KC-3 — Policy middleware is a `policy` package in core that wraps `llm.ChatModel`
as a capability-preserving decorator, mirroring `otelmodel.Wrap` (K3).** It is
*not* a hook system. It is *not* placed inside the agent loop — it lives at
the *model boundary* so every agent paradigm benefits without a code edit.
Public surface: `policy.Wrap(inner llm.ChatModel, gates ...Gate) llm.ChatModel`;
`type Gate interface{ Inspect(ctx, ev Event) error }` where `Event` is
typed-union `{Kind: PreGenerate|PostGenerate|PreStream|StreamDelta|PostStream, Req *Request, Resp *Response}`.
Built-in gates: `PIIRedactor` (regex), `InjectionScanner` (regex), `MaxInputLen`.
Sentinel: `policy.ErrBlocked` (wrapping the deciding gate). **Streaming case** —
`PreStream` runs once before the first delta; `StreamDelta` may run on each
text delta (best-effort, callers opt in via gate option, default off because
of cost); `PostStream` runs at `EventDone`. Composition with `otelmodel.Wrap`:
the documented stack is `policy.Wrap(otelmodel.Wrap(provider))` — outer-most
policy denies before observed; middle observes; inner-most makes the call.
Audit-log via optional `policy.OnDecision(fn func(Decision))` hook on the
wrapper — observation only, never interception.

**KC-4 — Budget/cancellation is ctx-keyed for *propagation* + a `budget.Tracker`
interface for *enforcement*.** Public surface: `budget.WithBudget(ctx, *Tracker) context.Context`;
`budget.From(ctx) *Tracker` (returns a no-op tracker if absent — safe to call);
`type Tracker interface{ Charge(Usage) error; Remaining() Budget }`; built-in
tracker `budget.NewStrict(b Budget) *Tracker` (deny on exhaustion);
`budget.NewSoft(b Budget) *Tracker` (warn-only via `OnExhausted` callback).
Wall-clock budget composes with stdlib — `budget.NewStrict` calls
`context.WithDeadline(ctx, time.Now().Add(b.Wall))` internally and propagates
that ctx. **Single integration chokepoint:** the `generateFromPrompt` helper
in `agents/` calls `budget.From(ctx).Charge(...)` before *and* after the LLM
call, so every paradigm inherits enforcement uniformly. Streaming: tracker
charges per `EventDone.Usage`; partial cancellation mid-stream surfaces as
`ctx.Err()` (the same as stdlib cancellation today). `MaxSteps` etc. are
**not deprecated** — per-agent step caps and cross-agent budgets coexist.
Cost is opt-in: core ships `Budget.Cost float64` and `Tracker.Charge` accepts
a `Cost` value; **core does not ship a provider→cost table**. Users wire
their own `CostMapper` (or pull it from the providers sister repo in a future
milestone).

**KC-5 — Breaking-change avoidance: every new surface is in a *new* package
or a *new* optional interface.** No edit to `llm.ChatModel`, `agents.Agent`,
`memory.Memory`, or `orchestrate.NodeFunc[S]`. New packages: `policy`,
`budget`, possibly `orchestrate.Supervisor` (as a type inside the existing
package — additive). New optional interfaces (asserted by callers): `Gate`
(in `policy`), `Tracker` (in `budget`). The v0.5.1 → v0.6.0 bump is **minor
(additive)**; existing v0.5.1 callers compile unchanged against v0.6.0.
**No `/v2` import path** — that's reserved for breaking changes (rag-style
v1.0 compatibility-promise discipline). v1.2 deliberately stays under that
ceiling.

## Proposed Phase Breakdown (Phases 35–37)

The global phase counter continues from 34 (v1.1 ended at Phase 34, expanded
to 9 slices).

### Phase 35 — Budget/cancellation plumbing
**Goal:** the `budget` package exists, propagates through `ctx`, enforces
token/call/wall-clock/cost budgets at the `generateFromPrompt` chokepoint;
every existing agent paradigm honors it with zero behavior change when no
budget is set.
- **35-01** — `budget` package skeleton: `Budget`, `Tracker`, `WithBudget`,
  `From`, `NewStrict`, `NewSoft`. ~150 LOC + tests using `ScriptedLLM`.
- **35-02** — integrate into `generateFromPrompt`: charge before + after
  every LLM call; honor `ctx.Err()` on exhaustion; preserve existing
  `MaxSteps`/`MaxTurns` semantics. Touches `agents.go`, `react.go`,
  `plan_solve.go`, `reflection.go`, `simple.go`, `function_call.go`.
- **35-03** — streaming integration: charge per `EventDone.Usage`; document
  the partial-stream cancellation semantics; add example
  (`examples/budget/`).

**Repos:** `llm-agent`
**Requirements covered:** `CORE-BUDGET-01..03` (TBD by REQUIREMENTS phase)
**Depends on:** none — v1.1 closed clean

### Phase 36 — Policy middleware (`policy` package + 3 built-in gates)
**Goal:** a `policy` package ships as a capability-preserving `llm.ChatModel`
decorator (mirrors `otelmodel.Wrap`); 3 built-in gates (PII redaction,
injection detection, max-input-length); audit log via `OnDecision`;
composes cleanly with `otelmodel.Wrap` (documented stack).
- **36-01** — `policy` package skeleton: `Wrap`, `Gate`, `Event`, `Decision`,
  `ErrBlocked`. Capability-preserving wrapper (handles `ToolCaller` /
  `Embedder` / `StructuredOutputs` assertions like `otelmodel.Wrap` does).
- **36-02** — 3 built-in gates: `PIIRedactor`, `InjectionScanner`,
  `MaxInputLen`. Patterns lifted from `llm-agent-rag/guard` where they
  overlap (but in core — separate file, no rag import).
- **36-03** — compose-with-otel integration test; documented stack;
  example (`examples/policy/`).

**Repos:** `llm-agent`
**Requirements covered:** `CORE-POLICY-01..03` (TBD)
**Depends on:** Phase 35 (gates can charge tokens for blocked requests)

### Phase 37 — Multi-agent coordination (`orchestrate.Supervisor`)
**Goal:** an iterative supervisor↔worker primitive that composes with
`StateGraph[S]`, honors budget + policy, and ships as
`orchestrate.Supervisor`.
- **37-01** — `Supervisor` design + skeleton: `NewSupervisor`,
  `SupervisorOptions{ Planner, Workers, MaxRounds, ParseDispatch,
  BuildAggregate }`. Implementation as a typed `StateGraph[S]` facade.
- **37-02** — integration with budget (37-01 honors `ctx`-keyed budget;
  rounds count against `Budget.Calls`) and policy (workers get
  policy-wrapped models by convention; documented in package doc).
- **37-03** — example (`examples/supervisor/` — a research-assistant
  demo, deterministic via `ScriptedLLM`); compose-with-`StateGraph`
  test (Supervisor inside StateGraph, StateGraph inside Supervisor).

**Repos:** `llm-agent`
**Requirements covered:** `CORE-COORD-01..03` (TBD)
**Depends on:** Phase 35, Phase 36

### Phase 38 — Milestone close (NOT a new feature phase)
**Goal:** core `v0.6.0` tagged; CHANGELOG entry; planning docs archived;
milestone audited.
