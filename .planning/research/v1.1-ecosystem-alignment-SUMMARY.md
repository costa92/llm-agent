## Executive Summary

`llm-agent-rag` reached a frozen, documented, gate-protected `v1.0.0` public API on
2026-05-21. The v1.0 milestone was deliberately scoped to that one repo (keystone
KS-1) — the core `llm-agent` module and the three sister repos were left on their
own version tracks. The result, six milestones later, is a **coherent SDK orbited
by four repos that all point at stale versions of it**. This is normal accumulated
drift, not a defect, but it is now the single highest-value piece of unscoped work:
the umbrella's headline claim — "four coordinated repos plus a standalone RAG SDK" —
is currently only half-true. They are coordinated at the *source* level (everything
still builds) but not at the *release* level (every `go.mod` lags).

The audit below confirms the good news first: **all five repos build and pass their
own test suites today**, and **every cross-repo bump I trial-ran (`core → rag v1.0.0`,
`otel → rag v1.0.0`, `customer-support → providers v0.1.1`) resolves and compiles**.
There is no architectural breakage. This is a *housekeeping and re-tagging*
milestone, not a redesign.

The bad news is concentrated and specific. The core repo's `go.mod` requires
`llm-agent-rag v0.1.4` — **eight minor versions and a major version stale** (current
is `v1.0.0`); its `go.sum` still pins only `v0.1.4`. When I trial-bumped core to
`v1.0.0`, the build stayed clean and — critically — stayed **stdlib-only** (`go list
-deps ./rag` pulls in *no* `pgx`/`pgvector`; the facade only touches stdlib-only rag
subpackages), but **7 facade tests fail with `rag: vector dimension mismatch`**. That
is a real behavior regression in the compatibility facade between rag v0.1.4 and
v1.0.0 and is the one piece of genuine *code* work in this milestone. The other three
repos are cleaner: `otel` bumps to rag v1.0.0 green (its `otelrag` package consumes
no API that changed v0.4→v1.0), `providers` is already healthy on `main`, and
`customer-support` builds green but consumes stale sister tags (`otel v0.1.0`,
`providers v0.1.0`) and is parked on a CI-fix branch.

Branch hygiene is the other theme. `otel` is on `feat/otelrag-wrap-rag-system` with
**4 unmerged commits — including the entire `otelrag` RAG-wrapping feature** — and
`customer-support` is on `fix/pr-governance-auto-merge-permissions` with 2 unmerged
CI-fix commits. Neither repo's `main` reflects its actual current state. The
milestone must land those branches before it can re-tag from `main`.

**Recommendation:** scope v1.1 as a tight 4-phase milestone (Phases 31–34) that does
exactly four things — fix the core facade against rag v1.0.0, merge the stranded
sister-repo branches, walk a coherent dependency-bump + re-tag wave across all four
repos, and harden the umbrella CI so this drift is caught next time. Explicitly
**out of scope**: the `llm-agent-rag` deployment layer (HTTP/CLI/caching),
productionizing the customer-support demo, incremental community maintenance, and
live-Postgres CI wiring. Those are real work but they are *features*; v1.1 is an
*alignment* milestone and must stay one.

## Recommended Milestone Scope

### In scope (v1.1)

1. **Repair the core `llm-agent` RAG facade against rag v1.0.0.** Bump the dep to
   `v1.0.0`, fix the 7 `vector dimension mismatch` facade-test failures by
   realigning the facade adapters (`rag/store.go`, `rag/embedder.go`, `rag/rag.go`)
   to the v1.0.0 SDK surface, refresh `go.sum`, confirm `rag/contract_test.go` still
   compiles, and **prove the facade stays stdlib-only** (`go list -deps ./rag`
   pulls no third-party module).
2. **Merge the two stranded sister-repo branches.** `otel`'s
   `feat/otelrag-wrap-rag-system` (4 commits — the whole `otelrag` feature) and
   `customer-support`'s `fix/pr-governance-auto-merge-permissions` (2 commits) must
   land on `main` so each repo can be re-tagged from a `main` that reflects reality.
3. **Walk a coherent dependency-bump + re-tag wave across all four repos**, in
   dependency order: core → rag-dep current, then `otel` (rag dep `v0.3.0 → v1.0.0`),
   `providers` (tag the untagged deepseek/minimax work), `customer-support`
   (`otel`/`providers` deps → the new tags). Land coordinated tags
   (proposed: core `v0.5.0`, otel `v0.2.0`, providers `v0.2.0`,
   customer-support `v0.2.0`).
4. **Harden the umbrella so this drift cannot silently recur.** Close the gap
   between `test.yml` (pins the stale dep) and `umbrella.yml` (checks out rag
   master): add a dependency-currency check / freshness gate, and prune the stale
   per-repo governance branches.

### Explicitly out of scope (non-goals)

| Non-goal | Why deferred |
|----------|--------------|
| `llm-agent-rag` deployment layer (HTTP service, CLI, caching) | A *feature* milestone, deferred since v0.6 by explicit operator choice. v1.1 is alignment, not feature growth. The first obvious post-alignment `v1.x` candidate |

| New non-stdlib dependency in core `llm-agent` | Absolute hard rule. The facade repair must be proven to keep core stdlib-only |

## Keystone Decisions (KE-1 .. KE-7) — for the roadmapper to ratify

**KE-1 — Scope is alignment, not features.** v1.1 brings the core + 3 sister repos
current with `llm-agent-rag v1.0.0` and to coherent tagged baselines. It ships **no
new feature** in any repo. The `otelrag` feature already exists in code (it merely
needs merging) — landing it is alignment, not new work.

**KE-2 — `llm-agent-rag` is the fixed point and is not touched.** Its frozen v1.0.0
API and `internal/apisnapshot`/`contract` gates are inviolate. Every other repo
aligns *to* rag; rag aligns to nothing. No rag re-tag in v1.1.

**KE-3 — The core RAG facade must be repaired *and proven stdlib-only*.** The
`vector dimension mismatch` regression is fixed inside the facade adapters — never
by adding a dep, never by changing rag. The phase's exit gate is: `go build ./...`
+ `go test ./...` green **and** `go list -deps ./rag` lists zero third-party
modules **and** `go.sum` contains only the `llm-agent-rag` line(s).

**KE-4 — Branches land before tags.** No repo is re-tagged from a `main` that does
not reflect its real state. `otel`'s `feat/otelrag-wrap-rag-system` and
`customer-support`'s fix branch are merged to `main` *first*; tags are cut from
`main` only.

**KE-5 — Coordinated tag wave, dependency-ordered, no `replace` directives.** Bumps
flow core → otel → providers → customer-support. Proposed tags: core `v0.5.0`,
otel `v0.2.0`, providers `v0.2.0`, customer-support `v0.2.0`. Tagged-release
branches carry **zero `replace` directives** (INFRA hard rule). rag's back-edge
`require llm-agent v0.4.0` may be bumped to core's new tag as optional housekeeping
or left as-is — bumping is *not* required (rag's frozen API is unaffected) and must
not trigger a rag re-tag.

**KE-6 — The umbrella CI gains a dependency-currency gate.** The root cause of this
drift is that `test.yml` tests against the pinned (stale) rag dep while
`umbrella.yml` checks out rag *master* — so a stale `go.mod` is invisible to the
per-repo CI. v1.1 adds a check (a `go list -m`-based freshness assertion, or an
umbrella step that fails when a sister `go.mod` lags the latest tag of a sibling)
so future drift fails CI instead of accumulating silently.

**KE-7 — Live-Postgres CI wiring stays deferred.** It is genuine carried-forward
debt, but it is an infra project of independent size; bundling it would break the
"tight alignment milestone" discipline. v1.1 records it as still-deferred; it is a
candidate for the milestone *after* v1.1.

## Proposed Phase Breakdown (Phases 31–34)

The global phase counter continues from 30 (v1.0 ended at Phase 30).

### Phase 31 — Core RAG facade re-alignment to `llm-agent-rag v1.0.0`
**Goal:** core `llm-agent` builds and tests green against rag `v1.0.0`, the facade
behaves identically to callers, and the core stays provably stdlib-only.
- **31-01** — Diagnose the `vector dimension mismatch`: diff the v0.1.4 → v1.0.0
  `store.InMemoryStore` / embedding dimension contract; pinpoint where the facade
  adapter (`rag/store.go`, `rag/embedder.go`, `rag/rag.go`) wires a mismatched dim.
- **31-02** — Bump `go.mod` to `llm-agent-rag v1.0.0`, refresh `go.sum`, repair the
  facade adapters so the dimension contract holds end-to-end; all 7 failing facade
  tests pass.
- **31-03** — Verify gate: `go vet ./... && go test ./...` green; `go list -deps
  ./rag` proves zero third-party modules; `go.sum` minimal; `rag/contract_test.go`
  cross-repo gate compiles. Update the facade `docs/` migration notes.

### Phase 32 — Sister-repo branch landing & hygiene
**Goal:** every sister repo's `main` reflects its true current state; stale branches
pruned; nothing tagged yet.
- **32-01** — `llm-agent-otel`: merge `feat/otelrag-wrap-rag-system` → `main` (the
  4-commit `otelrag` feature); confirm `go build ./... && go test ./...` green on
  `main`; prune stale `docs/`/`fix/` governance branches (local + origin).
- **32-02** — `llm-agent-customer-support`: merge
  `fix/pr-governance-auto-merge-permissions` → `main` (2 CI-fix commits); confirm
  build/test green on `main`; prune stale branches.
- **32-03** — `llm-agent-providers`: confirm `main` is clean and the untagged
  deepseek/minimax adapter work is present and green (no merge needed — this repo is
  already on `main`); branch-hygiene pass.

### Phase 33 — Coordinated dependency-bump & re-tag wave
**Goal:** all four repos consume current sibling tags; coordinated stable tags cut
in dependency order.
- **33-01** — Re-tag core `llm-agent v0.5.0` from `main` (the v1.0.0-aligned facade).
- **33-02** — `llm-agent-otel`: bump `llm-agent-rag v0.3.0 → v1.0.0` (optionally
  `llm-agent → v0.5.0`), refresh `go.sum`, verify build/test green, tag `v0.2.0`.
- **33-03** — `llm-agent-providers`: optionally bump `llm-agent → v0.5.0`, verify,
  tag `v0.2.0` (captures the deepseek/minimax adapters).
- **33-04** — `llm-agent-customer-support`: bump `llm-agent-otel → v0.2.0`,
  `llm-agent-providers → v0.2.0` (optionally `llm-agent → v0.5.0`), refresh
  `go.sum`, verify build/test green, tag `v0.2.0`. Confirm no `replace` directives
  in any tagged branch.

### Phase 34 — Umbrella coherence gate & milestone close
**Goal:** dependency drift fails CI in future; the umbrella is provably coherent;
milestone audited and closed.
- **34-01** — Add a dependency-currency gate: a CI step (umbrella or per-repo) that
  fails when a sister `go.mod` pins a sibling version older than that sibling's
  latest tag. Close the `test.yml` vs `umbrella.yml` blind spot.
- **34-02** — Full umbrella verification: a clean 5-repo `go work` build + test pass
  (CI-style, `GOWORK=off` for per-repo, `go work` only for the umbrella job) proving
  every repo resolves the coordinated tag set.
- **34-03** — Milestone audit + close: `v1.1-MILESTONE-AUDIT.md`, planning-doc
  updates, archive ROADMAP/REQUIREMENTS, record the coordinated tag set.

**Ordering rationale:** Phase 31 is first because the core facade is the only real
*code* fix and everything downstream depends on a green core. Phase 32 (branch
landing) must precede Phase 33 (tagging) — KE-4: never tag a misleading `main`.
Phase 33 walks the bump wave strictly in dependency order (core → otel → providers
→ customer-support) so each repo bumps against an already-tagged sibling — no
`replace` directives needed. Phase 34 last: the coherence gate can only be written
once the coherent state exists to assert against.
