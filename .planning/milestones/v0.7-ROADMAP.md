## Milestone v0.7: GraphRAG — relationship-traversal retrieval

**Goal**: add Tier-1 lightweight GraphRAG to `llm-agent-rag` — extract a
knowledge graph (entities + typed relations) from ingested documents and
retrieve by traversing it, fused as a fourth signal alongside the existing
dense / lexical / structure retrieval. LightRAG-end of the spectrum;
Microsoft-GraphRAG community detection and global search are deferred to
v0.8. The milestone is additive — no existing seam changes shape, and no new
module dependency (no graph database).

**Repos**: `llm-agent-rag` (only)

**Requirements in scope**: `RAG-GRAPH-01..06`

**Keystone decisions** (ratified from `.planning/research/v0.7-graphrag-SUMMARY.md`):

- **KG-1** — v0.7 targets Tier-1 lightweight GraphRAG; community detection /
  global search are v0.8.
- **KG-2** — the graph is a `store.GraphStore` optional capability
  (mirroring `store.LexicalSearcher`): stdlib in-memory impl + `postgres`
  recursive-CTE impl. No graph database; `GraphStore` stays an interface so
  Neo4j is a clean future seam point.
- **KG-3** — extraction is dual-mode: an `EntityExtractor` seam with an LLM
  impl (scripted-model tested) and a deterministic non-LLM impl. The default
  path stays zero-LLM-capable.
- **KG-4** — `GraphRetriever` fuses as a fourth RRF signal; it never
  replaces dense/lexical retrieval. Default `rag.System` behavior is
  unchanged unless graph retrieval is explicitly wired.
- **KG-5** — incremental graph reconciliation on re-ingest is in scope from
  day one (provenance-based removal + union-merge), not a follow-up.
- **KG-6** — canonicalization is deterministic exact-match only; fuzzy /
  embedding entity resolution is a documented limitation deferred to v0.8.
- **KG-7** — traversal is hard-bounded: depth cap 2 (default 1) plus a
  per-hop fan-out cap, enforced in both the in-memory walk and the
  recursive CTE.

## Phases

### Phase 20: Knowledge-graph construction — complete 2026-05-18

**Goal**: extract a canonicalized entity/relation graph from ingested
documents — an `EntityExtractor` seam with an LLM impl and a deterministic
non-LLM impl, wired as a post-split ingest stage.

**Requirements covered**: `RAG-GRAPH-01`, `RAG-GRAPH-02`

- `20-01` new `graph` package: `Entity`/`Relation`/`Graph` + `EntityExtractor`
  seam; `LLMEntityExtractor` over `generate.Model` (pipe-delimited prompt +
  lenient line parser), scripted-model unit tests covering malformed output.
- `20-02` deterministic `DictionaryEntityExtractor` (zero-LLM path);
  `Canonicalize` (exact-match `(name, type)` merge + provenance + endpoint
  resolution); graph extraction wired into `Import` as a post-split stage,
  surfaced on `ImportResult.Graph`.

### Phase 21: Graph storage — complete 2026-05-18

**Goal**: persist and traverse the graph behind a `store.GraphStore`
optional capability — stdlib in-memory and `postgres` recursive-CTE
implementations — with hard-bounded traversal and incremental reconciliation
on re-ingest.

**Requirements covered**: `RAG-GRAPH-03`, `RAG-GRAPH-04`

- `21-01` `store.GraphStore` optional-capability interface + `graph.Subgraph`;
  pure-stdlib in-memory adjacency implementation (`UpsertGraph` union-merge,
  `RemoveGraphBySource` provenance GC, depth/fan-out-bounded `Neighborhood`);
  `storetest.RunGraphConformance`.
- `21-02` `postgres` `GraphStore`: `entities`/`relations` tables in
  `Migrate()`, recursive-CTE `Neighborhood`; `RunGraphConformance` wired for
  postgres, env-gated.
- `21-03` `Import` persists the graph via `UpsertGraph`; `ReplaceSource`
  re-ingest calls `RemoveGraphBySource` first — provenance-based
  reconciliation, not append.

### Phase 22: Graph-traversal retrieval and fusion — complete 2026-05-18

**Goal**: retrieve via the graph — query entity linking, bounded
neighborhood expansion, proximity-decay scoring — and fuse it as a fourth
RRF signal, with eval coverage.

**Requirements covered**: `RAG-GRAPH-05`, `RAG-GRAPH-06`

- `22-01` `EntityLinker` seam + `LexicalEntityLinker`; `retrieve.GraphRetriever`
  — entity linking → bounded `Neighborhood` → proximity-decay-scored hits +
  `GraphTrace`.
- `22-02` fuse graph as a fourth RRF signal in `HybridRetriever`
  (`Graph` field, `FusionAttribution.GraphRank`, `EnableGraph` toggle);
  graph attribution in `retrieve.Trace` + `rag.Diagnostics`.
- `22-03` `eval` graph-on/off A/B; deterministic worked example;
  `docs/graphrag.md` with the v0.8 deferral list.

## Outcome

All three phases executed and verified green; milestone audit PASS (6/6
requirements, KG-1..KG-7 honored). `llm-agent-rag` gained **no new
dependency** — the `graph` package, `store.GraphStore` (in-memory +
postgres recursive-CTE), and `retrieve.GraphRetriever` are all stdlib plus
existing seams. `llm-agent-rag` tagged `v0.4.0`.

Carried-forward debt: the `postgres` graph path is env-gated and not yet
verified against a live database — the same standing as the v0.5 `tsvector`
debt.
