# Requirements: v1.0 — API stabilization and the compatibility promise

**Defined:** 2026-05-20
**Core Value:** the core `llm-agent` module stays stdlib-only and zero-dep;
`llm-agent-rag` — mature after six milestones — locks its public API,
documents a Go-module compatibility promise, and tags `v1.0.0`.

## Milestone Scope

v1.0 is a **stabilization / quality milestone — no new features**.
`llm-agent-rag` has grown from `v0.1.0` to `v0.6.0` across the v0.5→v0.9
arc and now spans the full practical GraphRAG spectrum. v1.0 is where its
public surface stops changing: the API is audited, the accidental and
inconsistent parts are fixed *before the freeze*, the module commits to the
Go import-compatibility promise, and `llm-agent-rag` is tagged `v1.0.0`.

**Scope is `llm-agent-rag` v1.0.0 only** (KS-1). The core `llm-agent`
module and the sister repos (`llm-agent-otel`, `llm-agent-providers`,
`llm-agent-customer-support`) stay on their own version tracks; the
existing `contract` package is the documented, gated cross-repo seam.

The milestone is an **audit → fix → verify-and-tag** arc. The only code
changes are pre-freeze naming/consistency fixes, doc comments, the
API-snapshot gate, and stale-doc corrections — `llm-agent-rag` is already a
clean codebase (no `TODO`/`replace`/dead code).

Reference: `.planning/research/v1.0-api-stabilization-SUMMARY.md` — the API
audit and the KS-1..KS-8 keystone decisions this milestone ratifies.

## v1.0 Requirements

### API audit and pre-freeze decisions

- [ ] **RAG-API-01**: a written, reviewed inventory of `llm-agent-rag`'s
      entire exported surface (all 19 importable packages + the
      build-tagged `adapter/llmagent`) classifies every exported symbol as
      keep / rename / unexport; it confirms in writing that there are no
      accidental exports and records the `Ask`/`AskGlobal`/`AskDrift` vs
      `*Options` naming as a ratified deliberate decision.
- [ ] **RAG-API-02**: the ratified pre-freeze breaking fixes are applied —
      `eval.Evaluator`→`RetrievalEvaluator` and `eval.Result`→`RetrievalResult`
      (symmetry with the prefixed answer-path evaluators), with all call
      sites and `examples/` updated; the `ragkit` root package comment is
      rewritten to document it as a deliberate doc-anchor; and the stale
      README "Not implemented yet" claims are corrected. No naming change
      is possible after the `v1.0.0` tag.

### Documentation and the compatibility policy

- [ ] **RAG-API-03**: every importable package has a package doc comment
      (11 are currently missing, including `rag`, the front door); every
      exported type, function, method, and `Err*` value has a
      name-prefixed doc comment — a `go doc`-driven gap-fill sweep across
      every package.
- [ ] **RAG-API-04**: `docs/compatibility.md` states the Go
      import-compatibility rule (additive-only within v1, the
      interface-method gotcha called out), semver, the `/v2` import-path
      rule for breaking changes, the `contract`-package sub-contract, the
      `postgres` external-dependency policy, the `adapter/llmagent`
      coverage, and the minimum Go version floor; `README.md` is updated to
      a "stable, v1.0" status linking the policy.

### API-stability gate, freeze, and the v1.0.0 tag

- [ ] **RAG-API-05**: a full exported-surface snapshot gate ships — a
      pure-stdlib generator (`go doc` / `go/packages`-based) producing a
      committed baseline (`api/v1.snapshot.txt` or similar) plus a
      `go test`-time regeneration-diff that fails any PR renaming,
      removing, or re-signing an exported symbol; it complements the
      existing cross-repo `contract` gate. Wired into CI. No new
      dependency.
- [ ] **RAG-API-06**: `CHANGELOG.md` gains a `[v1.0.0]` entry framing the
      API freeze + compatibility promise and listing the Phase-28 renames
      as the final breaking changes; a final full-suite verification
      passes (`GOWORK=off go vet ./... && go test ./...`, plus
      `-tags llmagent`); `llm-agent-rag` is tagged `v1.0.0`.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Any new feature — retriever, answer path, eval metric, store backend | v1.0 is a freeze, not a feature milestone (KS-2) |
| An `eval` package redesign / unification refactor | v1.0 renames for consistency only — it does not redesign (KS-4) |
| The `llm-agent-rag` deployment layer (HTTP service, CLI, caching) | A deliberate non-goal; deferred since v0.6 |
| Incremental community maintenance, claim/covariate extraction, a graph database | v0.9 deferrals — stay deferred to a post-v1.0 `v1.x` (additive) or `/v2` (breaking) |
| Re-versioning the core `llm-agent` module or any sister repo | KS-1 — v1.0 scope is `llm-agent-rag` only |
| Performance optimization | v1.0 freezes the API, not the implementation |
| New dependencies; `apidiff` as a `go.mod` require | KS-8 — stdlib-only holds; the snapshot gate is `go doc`-based |
