<objective>
The `eval` package scores retrieval only â€” it has no measure of generation
quality. This slice adds the generation-side judge: a `Judge` seam that
scores an answer for **groundedness** (is it supported by the retrieved
context?) and **answer relevance** (does it address the query?), plus an
`LLMJudge` that implements the seam as an LLM-as-judge over `generate.Model`.
These are legs 2 and 3 of the RAG Triad.
</objective>

<context>
- `eval` package (`eval/eval.go`, `eval/loader.go`) currently scores
  retrieval. It already imports `rag` and `store`.
- `generate.Model` is the abstract generation seam:
  `Generate(ctx, generate.Request) (generate.Response, error)`,
  `Request{SystemPrompt, Messages []Message, Metadata}`,
  `Response{Text string}`.
- The project's determinism discipline: an LLM-backed type is unit-tested
  with a scripted model returning fixed output â€” never a live call.
- No new dependency: lenient JSON parsing is `encoding/json` + `strings`.
- See `16-RESEARCH.md` Decisions 1 and 2.
</context>

<tasks>
1. Create `eval/judge.go` with the seam:
   ```go
   // JudgeRequest is one answer to be scored against its query and the
   // context it was supposed to be grounded in.
   type JudgeRequest struct {
       Query   string
       Answer  string
       Context []string
   }

   // Judgement scores an answer. Groundedness and AnswerRelevance are in
   // [0,1]; higher is better.
   type Judgement struct {
       Groundedness    float64
       AnswerRelevance float64
       Rationale       string
   }

   // Judge scores a generated answer. Implementations may be heuristic or
   // model-backed; callers supply their own so eval stays vendor-neutral.
   type Judge interface {
       Judge(ctx context.Context, req JudgeRequest) (Judgement, error)
   }
   ```
2. Add `LLMJudge` to `eval/judge.go`:
   ```go
   type LLMJudge struct {
       Model generate.Model
   }
   ```
   - `Judge` errors if `Model == nil` (`ErrJudgeModelRequired`).
   - Build a `generate.Request`: a system prompt instructing the model to
     act as a strict RAG evaluator and reply with **only** a JSON object
     `{"groundedness":<0..1>,"answer_relevance":<0..1>,"rationale":"..."}`;
     a user message embedding the query, the answer, and the numbered
     context passages.
   - Call `Model.Generate`. Parse the response leniently: locate the first
     `{` and the last `}` and `json.Unmarshal` that substring into a struct
     with `groundedness` / `answer_relevance` / `rationale` JSON tags. A
     response with no JSON object â†’ return a descriptive error.
   - Clamp `Groundedness` and `AnswerRelevance` into `[0,1]` before
     returning (a model may emit 1.2 or -0.1).
3. Export `ErrJudgeModelRequired`.
4. Tests in `eval/judge_test.go`:
   - a scripted `generate.Model` returning a clean JSON judgement â†’ assert
     `LLMJudge` returns the parsed scores and rationale.
   - a scripted model returning JSON wrapped in prose ("Here is my
     evaluation: { ... } done") â†’ assert the lenient parser still extracts
     it.
   - a scripted model returning out-of-range scores (e.g. 1.5, -0.3) â†’
     assert the result is clamped to `[0,1]`.
   - a scripted model returning no JSON â†’ assert an error.
   - `LLMJudge{}` (nil model) â†’ assert `ErrJudgeModelRequired`.
5. Run every command in `<verify>`.

Out of scope:

- A heuristic (non-LLM) judge â€” `LLMJudge` is the deliverable; the seam

- Wiring a judge into `rag.System` â€” judging is an offline eval activity.
</tasks>
