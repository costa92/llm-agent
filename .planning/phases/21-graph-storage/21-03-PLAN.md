<objective>
Complete RAG-GRAPH-04: wire `Import` to persist the canonicalized graph
into a `store.GraphStore`-capable store, and reconcile the graph on a
`ReplaceSource` re-ingest — provenance-based removal of the old
contributions before the re-extracted subgraph is union-merged in.
</objective>

<context>
- 21-01 delivered `store.GraphStore` (`UpsertGraph` union-merge,
  `RemoveGraphBySource` provenance GC) and the in-memory implementation.
- `rag/import.go` `Import` (Phase 20): when `s.entityExtractor != nil` it
  extracts per chunk, `graph.Canonicalize`s, and sets
  `res.Graph = &g`. It already handles `ReplaceSource` for chunks — per
  document, when `opts.ReplaceSource && doc.SourceID != ""`, it calls
  `s.store.RemoveByFilter(...)` and accumulates `removedChunks`.
- `store.GraphStore` is an optional capability — `s.store` may or may not
  implement it; type-assert and degrade gracefully (the `LexicalSearcher`
  contract).
- See `21-RESEARCH.md` Decision 6.
</context>

<tasks>
1. In `rag/import.go` `Import`, after the canonicalized graph is built
   (`res.Graph` set) and after `s.store.Upsert` succeeds:
   - Type-assert `s.store` for `store.GraphStore`. If it does not
     implement it, or no `EntityExtractor` is configured, do nothing more
     — `res.Graph` is still returned (graceful degradation).
   - When the store is a `GraphStore` and an extractor ran:
     - **Re-ingest reconciliation:** for each document being imported with
       `opts.ReplaceSource && doc.SourceID != ""`, the document's prior
       chunks were just removed from the vector store via `RemoveByFilter`.
       Collect the chunk IDs that document contributes on *this* import and
       call `GraphStore.RemoveGraphBySource` for the **prior** chunk IDs of
       that source so stale entities/relations are GC'd before the merge.
       (Chunk IDs are derived from the document deterministically by the
       splitter; if prior chunk IDs are not recoverable, document the
       limitation — at minimum remove-by-source for the chunk IDs the
       splitter produces for the re-ingested document.)
     - Call `GraphStore.UpsertGraph(ctx, opts.Namespace, *res.Graph)` to
       union-merge the re-extracted subgraph.
   - A `GraphStore` error aborts the import with a wrapped error,
     consistent with `Upsert`/`embed` error handling.
2. Keep the change additive and ordered cleanly: vector `Upsert` first,
   then graph persistence — both inside `Import`, both failing the import
   loudly on error.
3. Tests in `rag/graph_test.go` (extend the Phase 20 file):
   - `Import` with `EntityExtractor` + the in-memory store (which is a
     `GraphStore`) → after import, the store's `Neighborhood`/`FindEntities`
     return the extracted entities (graph was persisted).
   - **Re-ingest reconciliation:** import a document, then re-import the
     same `SourceID` with `ReplaceSource: true` and changed content that
     drops an entity → the dropped entity is gone from the store's graph
     (GC'd), the surviving/added entities are present.
   - `Import` with an `EntityExtractor` but a store that is *not* a
     `GraphStore` → no error, `res.Graph` still populated (graceful
     degradation). Use a minimal `store.Store` wrapper that does not
     implement `GraphStore`, or document if the in-memory store cannot be
     un-graph-capable and assert the capability path instead.
4. Run every command in `<verify>`.

Out of scope:
- The `postgres` graph path's live verification — env-gated, carried debt.
- Graph retrieval — Phase 22.
</tasks>
