The defensive policy is: every repo gitignores `go.work`.

```bash
#!/usr/bin/env bash
# scripts/workspace.sh — write a sibling-aware go.work above this repo.
#
# Usage: run from any of the 4 sibling clones:
#   <parent>/llm-agent
#   <parent>/llm-agent-providers
#   <parent>/llm-agent-otel
#   <parent>/llm-agent-customer-support
#
# Result: <parent>/go.work points at all 4 modules. The file is gitignored
# in every repo (Pitfall 13). Idempotent — safe to re-run.
set -euo pipefail

script_dir="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
repo_dir="$(cd "$script_dir/.." && pwd)"
parent_dir="$(cd "$repo_dir/.." && pwd)"

cd "$parent_dir"

modules=()
for d in llm-agent llm-agent-providers llm-agent-otel llm-agent-customer-support; do
  if [ -d "$d" ] && [ -f "$d/go.mod" ]; then
    modules+=("./$d")
  fi
done

if [ "${#modules[@]}" -eq 0 ]; then
  echo "scripts/workspace.sh: no sibling Go modules found in $parent_dir" >&2
  exit 1
fi

# Drop any existing go.work first so re-runs are clean.
rm -f go.work go.work.sum

go work init "${modules[@]}"

echo "wrote $parent_dir/go.work pointing at:"
for m in "${modules[@]}"; do echo "  - $m"; done
echo
echo "go.work is .gitignored in every sibling repo. To revert: rm $parent_dir/go.work"
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `.gitignore` enforcement | Pure local-Git filter — once committed, every clone respects it. Cannot be bypassed without explicit `git add -f`. |
| `scripts/workspace.sh` execution | Run by the developer locally; `set -euo pipefail` makes failures loud; the script writes to the PARENT directory (above the repo) where any file write is highly visible. |
| `env: GOWORK: off` in CI | Pure GitHub Actions environment configuration; cannot be overridden mid-step (later steps inherit). The only way to defeat it is for a malicious PR to remove the env from the YAML — and that PR is itself caught by the `go mod tidy (drift check)` step (workflow self-validates). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-04-01 | Tampering | Accidentally committed `go.work` | mitigate | `.gitignore` lists `go.work` and `go.work.sum`; `git check-ignore -q go.work` returns 0 (Task 1 verify). Pitfall 13 prevention. |
| T-00-04-02 | Tampering | Workspace.sh malicious modification | mitigate | The script's `set -euo pipefail` + idempotent `rm -f go.work go.work.sum` design exposes any failure loudly. The script is identical across all 4 repos (INFRA-03) — a malicious change would require modifying 4 places. |
| T-00-04-03 | Information Disclosure | `go.work` content leaked into a public PR | accept | The script writes go.work in the PARENT directory (one level UP from the repo). Even if a developer accidentally `cd`s into a parent and runs `git status`, the file is OUTSIDE every repo's working tree — git does not see it. Acceptable. |
| T-00-04-04 | Elevation of Privilege | CI workflow privilege via env modification | mitigate | Workflow-level `env: GOWORK: off` is set at the top of the file before any step runs. A malicious PR cannot UNSET it for a single step (GitHub Actions evaluates workflow-level env first; step-level env can override per-step but cannot remove the workflow-level setting from earlier steps). The only escape is removing the env entirely — which is visible in the diff. |
| T-00-04-05 | Repudiation | `go.work` committed silently breaks downstream go get | mitigate | Combined with plan 00-05's release-precheck (which rejects `replace` directives on release/** branches), the policy is: gitignore prevents accidental commits; release-precheck catches deliberate ones. Layered defence. |
</threat_model>
