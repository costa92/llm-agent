<objective>
Add RAG-AGENT-01: a `MultiHopRetriever` decorator that decomposes a
compound query into sub-queries, retrieves for each through a wrapped
`Retriever`, and merges the sub-retrievals — with per-hop attribution in
the trace.
</objective>

<context>
- `retrieve.Retriever` — `Retrieve(ctx, Request) ([]store.Hit, Trace, error)`.
  `VariantRetriever{Base, Planner}` is the merge template: per query, call
  `Base.Retrieve`, dedup hits by `Chunk.ID` keeping the max score, preserve
  insertion order, sort/truncate to `req.TopK`. `ErrBaseRetrieverRequired`
  exists for a nil base. `appendUniqueStrings` merges trace string fields.
- `retrieve.Request` has `Query`, `TopK`, `QueryVariants`, routing fields.
- `retrieve.Trace` carries `OriginalQuery`, `QueryVariants`,
  `MatchedSections`, `ExpandedChunkIDs`, `SelectedChunkIDs`, etc.
- `retrieve` imports `generate` already (`LLMExpansionPreprocessor`).
- The project's tiering: a Noop/Heuristic/Model trio of implementations
  behind a seam (see `rerank`).
- See `19-RESEARCH.md` Decision 1.
</context>

<tasks>
1. Create `retrieve/multihop.go` (package `retrieve`):
   - `QueryDecomposer` interface:
     ```go
     type QueryDecomposer interface {
         Decompose(ctx context.Context, query string) ([]string, error)
     }
     ```
   - `HopAttribution struct { SubQuery string; HitCount int }`.
   - `HeuristicDecomposer struct{}` — `Decompose` splits `query` on `" and "`
     (case-insensitive), trims each part, drops parts shorter than 3 runes,
     dedups; returns `[]string{query}` when the split yields 0 or 1 part.
     No model — deterministic.
   - `LLMDecomposer struct { Model generate.Model; MaxSubQueries int }` —
     `Decompose`: if `Model == nil` return `[]string{query}`. Otherwise
     prompt the model (system prompt: "break a compound question into
     independent sub-questions, one per line; if it is already a single
     question, return it unchanged"). Parse the response leniently: split
     on newlines, trim, strip leading list markers (`-`, `*`, `1.`), drop
     empties. Cap the result at `MaxSubQueries` (`<= 0` → default 5). An
     empty parse → `[]string{query}`.
   - `MultiHopRetriever struct { Base Retriever; Decomposer QueryDecomposer }`
     implementing `Retriever`:
     - nil `Base` → `ErrBaseRetrieverRequired`; nil `Decomposer` →
       `HeuristicDecomposer{}`.
     - `Decompose` `req.Query`; empty result → `[]string{req.Query}`.
     - For each sub-query: a sub-request copy with `Query` set and
       `QueryVariants` cleared; call `Base.Retrieve`; append a
       `HopAttribution{SubQuery, HitCount: len(hits)}`.
     - Merge hits with the `VariantRetriever` pattern: dedup by `Chunk.ID`,
       keep the max score, preserve first-seen order; then sort by score
       descending and truncate to `req.TopK` (when `TopK > 0`).
     - Build the returned `Trace`: start from hop 1's sub-trace (carries
       routing/section data), set `OriginalQuery = req.Query`,
       `Hops = <attributions>`, `SelectedChunkIDs = <merged result IDs>`,
       and union `MatchedSections`/`ExpandedChunkIDs` across sub-traces via
       `appendUniqueStrings`.
2. Add `Hops []HopAttribution` to `retrieve.Trace` in `retrieve/retrieve.go`.
3. Tests `retrieve/multihop_test.go`:
   - `HeuristicDecomposer.Decompose` on `"capital of France and population
     of France"` → 2 sub-queries; on `"what is the capital of France"` → 1
     (the query itself).
   - `LLMDecomposer.Decompose` with a scripted `generate.Model` returning a
     two-line response → 2 sub-queries; with `Model == nil` → `[query]`.
   - `MultiHopRetriever.Retrieve` with a stub `Retriever` Base (returns
     canned hits keyed by sub-query) and `HeuristicDecomposer`: a compound
     query → both sub-queries hit the Base, hits are merged and deduped by
     `Chunk.ID`, `Trace.Hops` has 2 entries with correct `HitCount`s, the
     result respects `req.TopK`.
   - `MultiHopRetriever.Retrieve` with a nil `Base` → `ErrBaseRetrieverRequired`.
4. Run every command in `<verify>`.

Out of scope:
- The self-correcting loop (`agentic` package) — that is 19-02.
- Wiring `MultiHopRetriever` as a default in `rag.New` — it stays an opt-in

- Cross-hop reasoning (feeding hop-1 results into hop-2 queries) — v0.6
  multi-hop is parallel decomposition + merge, not sequential reasoning.
</tasks>
