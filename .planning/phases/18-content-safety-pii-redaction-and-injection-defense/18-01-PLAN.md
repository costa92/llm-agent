<objective>
The RAG pipeline ingests document content verbatim — PII flows into chunks,
vectors, and the store. This slice adds RAG-SEC-01: a new `guard` package
with a `Redactor` seam and a configurable rule-based `PIIRedactor`, wired
into `Import` so PII is redacted **before** chunking and embedding.
</objective>

<context>
- `rag/import.go` `Import` loops documents: `splitter.Split(doc, maxChars)`
  then embeds each chunk. Redaction must run on `doc.Content` before
  `Split`.
- `rag/options.go` — `Options` struct (no security fields yet).
- `rag/system.go` — `System` struct + `New`.
- `rag/observer.go` — `ImportTrace` (already has `Metrics obs.Metrics`).
- `ingest/types.go` — `ImportResult` (already has `Metrics obs.Metrics`).
- `guard` will be a new leaf package (imports only `regexp`/`strings`);
  `ingest` and `rag` may import it with no cycle.
- See `18-RESEARCH.md` Decisions 1 and 2.
</context>

<tasks>
1. Create `guard/redact.go` (package `guard`):
   ```go
   type Redaction struct { Kind string; Count int }
   type RedactResult struct { Text string; Redactions []Redaction }
   type Redactor interface { Redact(text string) RedactResult }

   type Rule struct {
       Kind        string
       Pattern     *regexp.Regexp
       Placeholder string
   }
   type PIIRedactor struct { Rules []Rule }
   ```
   - `PIIRedactor.Redact(text)` applies each rule in order: count matches
     (`Pattern.FindAllString`), replace with `Placeholder`. Return
     `RedactResult` with the redacted text and one `Redaction{Kind, Count}`
     per rule that matched at least once (skip zero-count rules). The zero
     `PIIRedactor` (no rules) redacts nothing.
   - `NewPIIRedactor() PIIRedactor` — built-in rules, each collapsing to a
     `[REDACTED:<KIND>]` placeholder:
     - `email` — standard address pattern.
     - `phone` — 10+ digit sequences allowing spaces/dashes/parens and an
       optional `+` country prefix.
     - `credit_card` — 13–16 digit sequences allowing spaces/dashes.
     - `ssn` — US `\d{3}-\d{2}-\d{4}`.
     - `ipv4` — dotted-quad.
   - Order rules so the more specific patterns (ssn) run before broader
     numeric ones (credit_card, phone) to avoid a broad rule consuming a
     specific match first.
2. Add `Redactor guard.Redactor` to `rag.Options` (`rag/options.go` —
   add the `guard` import).
3. Add a `redactor guard.Redactor` field to `System` (`rag/system.go`) and
   set it from `opts.Redactor` in `New` (no default — nil means off).
4. In `rag/import.go` `Import`, before `splitter.Split`: if
   `s.redactor != nil`, call `s.redactor.Redact(doc.Content)`, replace
   `doc.Content` with the result text, and accumulate the `Redaction`s
   across all documents (sum counts per `Kind`).
5. Add `Redactions []guard.Redaction` to `ingest.ImportResult`
   (`ingest/types.go` — add the `guard` import) and to `rag.ImportTrace`
   (`rag/observer.go`). Populate both from the accumulated per-kind tallies.
6. Tests `guard/redact_test.go`:
   - `NewPIIRedactor().Redact` on text containing an email, a phone number,
     a credit-card number, an SSN, and an IPv4 → assert each is replaced by
     its placeholder and the `Redactions` counts are correct.
   - text with no PII → `RedactResult.Text` unchanged, `Redactions` empty.
   - a custom `PIIRedactor{Rules: []Rule{...}}` with one caller-supplied
     rule → asserts configurability.
7. Tests in `rag` (e.g. `rag/redact_test.go`, package `rag`):
   - `Import` with `Options{Redactor: guard.NewPIIRedactor()}` of a document
     whose content contains an email → after import, retrieving the chunk
     shows the email replaced by `[REDACTED:EMAIL]`, and
     `ImportResult.Redactions` records the email count.
   - `Import` with no redactor → content is stored verbatim (opt-in proof).
8. Run every command in `<verify>`.

Out of scope:
- The injection scanner, `Neutralize`, sanitize wiring — that is 18-02.
- Redacting `doc.Title` or metadata — content only.
- A model-based PII detector — `PIIRedactor` is rule-based by design.
</tasks>
