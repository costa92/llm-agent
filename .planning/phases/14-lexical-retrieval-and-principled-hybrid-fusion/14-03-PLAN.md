<objective>
`HybridRetriever` already fuses dense, lexical, and structure results with
Reciprocal Rank Fusion â€” but the RRF constant `60` is hardcoded and the
fusion is opaque: the merged `Trace` shows none of the per-signal ranks or
RRF contributions. A caller cannot tell *why* a chunk surfaced.

This slice makes the RRF constant configurable (default unchanged) and
exposes per-signal fusion attribution in the retrieval `Trace`, satisfying
RAG-RETR2-02's "principled fusion with per-signal score attribution."
</objective>

<context>
- `HybridRetriever{Dense, Lexical, Structure Retriever}` in
  `retrieve/retrieve.go`. `Retrieve` runs each sub-retriever, then applies
  `rrfScores[id] += 1.0/float64(i+1+60)` per signal and sorts by the summed
  RRF score. The `60` is the only fusion knob and it is a literal.
- The merged `Trace` it returns carries only legacy section fields
  (`SearchPath`, `MatchedSections`, `ExpandedSections`, `SelectedChunkIDs`).
  Nothing records which signal contributed which rank.
- `Trace` is defined in `retrieve/retrieve.go`. Adding fields is additive and
  safe; the Phase 13 `contract` package pins the consumed surface â€” a new
  exported field should not break it, but the contract test is part of
  `<verify>` here to confirm.
- RRF with constant `k`: a chunk at rank `r` (0-based) for a signal
  contributes `1/(k + r + 1)`. Default `k = 60` reproduces the current
  `i+1+60` expression exactly.
- See `14-RESEARCH.md` Decision 4.
</context>

<tasks>
1. Add `RRFConstant float64` to `HybridRetriever`. A helper resolves it:
   zero â†’ `60`. Replace the literal `60` in the RRF expression with the
   resolved constant. With the default, fused scores are bit-for-bit identical
   to today.
2. Add the attribution types to `retrieve/retrieve.go`:
   ```go
   // FusionAttribution records how each retrieval signal ranked one chunk.
   // A rank of 0 means the signal did not return the chunk.
   type FusionAttribution struct {
       ChunkID       string
       DenseRank     int
       LexicalRank   int
       StructureRank int
       RRFScore      float64
   }
   ```
   Add `Fusion []FusionAttribution` to `Trace`.
3. In `HybridRetriever.Retrieve`, while applying RRF, record each signal's
   1-based rank per chunk ID (dense, lexical, structure). After fusion, build
   one `FusionAttribution` per fused chunk with the recorded ranks (0 where a
   signal did not return it) and the summed `RRFScore`. Sort `Trace.Fusion`
   by `RRFScore` descending (tie-break by `ChunkID` for determinism) and set
   it on the merged trace. Existing trace fields are untouched.
4. Tests in `retrieve/retrieve_test.go`:
   - **multi-signal boost** â€” a chunk returned highly by both dense and
     lexical ends up with a higher `RRFScore` (and earlier `Trace.Fusion`
     position) than a chunk returned highly by only one signal.
   - **attribution correctness** â€” for a known fixture, `Trace.Fusion`
     entries carry the exact per-signal ranks, with `0` for signals that did
     not return the chunk.
   - **default preserves behavior** â€” a `HybridRetriever` with `RRFConstant`
     unset produces the same hit ordering and scores as before this slice
     (assert against an explicit `RRFConstant: 60`).
   - **configurability** â€” a deliberately small `RRFConstant` changes the
     fused ordering in the expected direction.
5. Run every command in `<verify>`, including the cross-repo contract gate.

Out of scope:
- Weighting signals unequally (RRF is rank-based and weightless by design);
  per-signal weights are a possible future knob, not this slice.
- Surfacing `Trace.Fusion` through the core `llm-agent/rag` facade â€” the data
  exists on the standalone trace; facade exposure can follow when a consumer
  needs it.
- Changing lexical or dense scoring (14-01 / 14-02).
</tasks>
