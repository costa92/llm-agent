<objective>
`retrieve.LexicalRetriever` currently scores chunks with `lexicalScore()` â€”
binary token overlap (`+1` per distinct query token present). This ignores
term frequency, term rarity, and document length, so it ranks a passing
mention equal to a focused passage and lets stopword-like common terms
dominate.

This slice replaces that with Okapi BM25 â€” the standard lexical ranking
function â€” for the in-memory retrieval path, and introduces an optional
`store.LexicalSearcher` capability so backends that rank natively (Postgres,
landing in 14-02) are used instead of an in-process scan.
</objective>

<context>
- `LexicalRetriever{Store store.Store}` lives in `retrieve/retrieve.go`. Its
  `Retrieve` lists the namespace's chunks and scores each via `lexicalScore`
  (the token-overlap function, ~lines 1322-1341). `tokenize()` (~line 1310)
  is the shared tokenizer â€” reuse it so query and document tokenization match.
- `store.Store.Search` is vector-only. `store.Hit{Chunk StoredChunk, Score
  float64}`. `StoredChunk.Content` holds the raw text.
- BM25 needs corpus-wide stats (document count `N`, per-term document
  frequency `df`, average document length `avgdl`). Compute them over the
  chunks returned by `store.List(namespace, â€¦)` â€” for the in-memory store
  that is the entire namespace, so this is exact BM25, not an approximation.
- The optional-capability pattern already exists in this codebase
  (`RemoveByFilter`-style additions). `LexicalSearcher` follows it: a small
  interface a store *may* implement, type-asserted at the call site. This
  keeps `store.Store` itself unchanged.
- See `14-RESEARCH.md` Decision 1 and Decision 3 for the formula and the
  interface rationale.
</context>
