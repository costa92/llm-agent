<objective>
14-01 made the in-memory lexical path real BM25 and defined the optional
`store.LexicalSearcher` capability. This slice implements that capability on
`postgres.Store` so the persistent backend ranks keyword queries with
PostgreSQL's native full-text engine instead of forcing an in-process scan of
every row.
</objective>

<context>
- `postgres.Store` (`postgres/postgres.go`) wraps a `*pgxpool.Pool`. `Migrate()`
  creates the `chunks` table with `content TEXT NOT NULL` and an `embedding
  vector(N)` column; vector `Search` uses `embedding <=> $N`. There is no
  text index today.
- `store.LexicalSearcher` (from 14-01): `LexicalSearch(ctx, store.Query)
  ([]store.Hit, error)`. `LexicalRetriever` already type-asserts for it, so
  implementing it here is all the wiring this path needs.
- Postgres full-text search: a `tsvector` of the content, a GIN index on it,
  `websearch_to_tsquery` to parse user query text, `content_tsv @@ query` to
  match, `ts_rank_cd` to rank. See `14-RESEARCH.md` Decision 2.
- `ts_rank_cd` is a coverage-density rank, not BM25. That per-backend
  difference is acceptable — the contract is "relevance-ordered lexical hits,"
  not score parity across backends (vector backends already differ in
  distance metrics).
- `scanChunk()` already maps a result row to `store.StoredChunk`; the existing
  `Search`/`List` queries show the namespace + JSONB metadata + security
  filter SQL idiom to reuse.
</context>

<tasks>
1. `postgres.Config`: add `TextSearchConfig string`. Empty resolves to
   `"english"` at use sites (add a small `textSearchConfig()` accessor).
2. `Migrate()`: after the `chunks` table exists, add the full-text column and
   index idempotently:
   ```sql
   ALTER TABLE chunks ADD COLUMN IF NOT EXISTS content_tsv tsvector
     GENERATED ALWAYS AS (to_tsvector('<cfg>', content)) STORED;
   CREATE INDEX IF NOT EXISTS chunks_content_tsv_idx ON chunks USING GIN (content_tsv);
   ```
   The text-search config is fixed at column-definition time — interpolate
   `textSearchConfig()` into the DDL string (it is config, not user input;
   still, validate it against a small allowlist or `^[a-z_]+$` to be safe).
3. Implement `LexicalSearch(ctx context.Context, q store.Query) ([]store.Hit,
   error)`:
   - build `websearch_to_tsquery($cfg, $query)`; `WHERE content_tsv @@ query
     AND namespace = $ns` plus the same caller-filter + security-filter SQL
     `Search` applies.
   - `SELECT <chunk columns>, ts_rank_cd(content_tsv, query) AS score …
     ORDER BY score DESC LIMIT $topk`.
   - scan rows via `scanChunk` + score into `[]store.Hit`.
   - empty/whitespace query → empty result, no error.
4. In `store/storetest/storetest.go`, add `RunLexicalConformance(t *testing.T,
   s store.Store)`:
   - skip (via `t.Skip`) if `s` does not implement `store.LexicalSearcher`.
   - subtests: keyword match returns the relevant chunk; a term-frequency-heavy
     chunk ranks above a single-mention chunk; namespace isolation (a hit from
     another namespace never appears); security-filter trimming is enforced;
     empty query yields no hits.
5. Wire `RunLexicalConformance` into the env-gated Postgres test
   (`postgres/*_test.go`, alongside the existing `RunConformance` call) so it
   runs only when the live-Postgres env var is set. The in-memory store does
   not implement `LexicalSearcher`, so its conformance run simply skips the
   lexical suite.
6. Run every command in `<verify>`. The live-Postgres subtests are env-gated;
   if no database is reachable, state that explicitly in the SUMMARY rather
   than claiming they passed.

Out of scope:
- Changing the in-memory lexical path (done in 14-01).
- Hybrid fusion (14-03).
- Synonyms, custom dictionaries, or trigram/`pg_trgm` fuzzy matching.
</tasks>
