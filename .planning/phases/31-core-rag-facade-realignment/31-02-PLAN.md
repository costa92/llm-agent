<objective>
Repair the core `rag/` facade so it works against `llm-agent-rag v1.0.0`:

<context>
- 31-01 bumped `go.mod` to `v1.0.0` and recorded the precise diagnosis in
  `31-01-SUMMARY.md` — **read it first**; it lists the exact failing
  tests, each failure's code path, and whether any second-order break
  exists beyond the `nil`-vector hack.
- The bug: `storeAdapter.List` (`rag/rag.go`) emulates enumeration with
  `a.inner.Search(ctx, nil, stats.Count)`; v1.0.0's
  `store.InMemoryStore.Search` rejects the length-0 `nil` vector with
  `ErrDimensionMismatch`. The `storeAdapter.Search` namespace/filter
  branches (`pool = a.inner.Stats().Count`) feed the same hazard.
- Recommended fix (`31-RESEARCH.md` Decision 1), to confirm against
  31-01's findings:
  - **An optional `lister` interface + a real `List`.** The facade's
    `*InMemoryStore` (`rag/store.go`) wraps `ragstore.InMemoryStore`,
    which has the SDK's real `List`. Add an additive method on the facade
    `*InMemoryStore` that enumerates real chunks (delegating to the inner
    SDK store's `List`, or a `Stats`-bounded id walk). Define an
    unexported optional interface; have `storeAdapter.List` **type-assert**
    `a.inner` to it and use the real enumeration when present.
  - **Fallback** for a custom `VectorStore` lacking the optional method:
    `storeAdapter` sees every chunk through `Upsert`/`Remove` — maintain
    an internal id index and enumerate via `a.inner.Get(id)`. This keeps
    namespace/filter ops working for ANY `VectorStore`.
  - **Do NOT** add `List`/enumerate to the public `VectorStore` interface
    — that breaks external implementers. The optional-interface escape +
    the id-index fallback avoid it.
- KE-3: never fix by adding a dependency, never by changing `llm-agent-rag`.
- See `31-RESEARCH.md` Decision 1 + `31-01-SUMMARY.md`.
</context>
