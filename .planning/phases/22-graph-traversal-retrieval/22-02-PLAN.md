<objective>
Fuse graph retrieval into `HybridRetriever` as a fourth RRF signal —
alongside dense, lexical, and structure — gated by an `EnableGraph` toggle,
with the graph's contribution attributed in the trace and surfaced through
`rag.Diagnostics`. Completes RAG-GRAPH-05.
</objective>

<context>
- `retrieve.HybridRetriever{Dense, Lexical, Structure Retriever;
  RRFConstant float64}` fuses signals: an `apply(hits, ranks)` closure adds
  `1/(k+rank)` per signal; `FusionAttribution{ChunkID, DenseRank,
  LexicalRank, StructureRank, RRFScore}` records per-signal ranks. RRF is
  signal-count-agnostic.
- `Structure` is gated by `req.EnableStructure` — the pattern for
  `EnableGraph`. `retrieve.Request` carries the `Enable*` flags;
  `rag.SearchOptions` mirrors them and `rag/retrieve.go` maps
  `SearchOptions` → `retrieve.Request`.
- 22-01 delivered `GraphRetriever` (a `retrieve.Retriever`) and
  `GraphTrace`.
- `retrieve.Trace` carries `Fusion`; `rag.Diagnostics` surfaces retrieval
  attribution (`AutoRouteCandidates`, `RoutePolicy`, etc.).
- See `22-RESEARCH.md` Decisions 3 and 4.
</context>

<tasks>
1. In `retrieve/retrieve.go`:
   - add `Graph Retriever` to `HybridRetriever`.
   - add `GraphRank int` to `FusionAttribution`.
   - add `EnableGraph bool` to `retrieve.Request`.
   - add `Graph GraphTrace` to `retrieve.Trace`.
   - in `HybridRetriever.Retrieve`: when `req.EnableGraph && r.Graph != nil`,
     run `r.Graph.Retrieve(ctx, req)`, `apply(graphHits, graphRank)` into
     the RRF fusion (same `1/(k+rank)`), set `FusionAttribution.GraphRank`,
     and copy the graph sub-trace's `Graph` block onto the returned `Trace`.
     When graph is off or unset, behavior is byte-for-byte the prior
     behavior.
2. In `rag/options.go`: add `EnableGraph bool` to `SearchOptions`.
3. In `rag/retrieve.go`: map `SearchOptions.EnableGraph` →
   `retrieve.Request.EnableGraph` (alongside the other `Enable*` fields).
4. Surface graph attribution on `rag.Diagnostics` (`rag/system.go` +
   `rag/ask.go`): add a `GraphTrace`-shaped field to `Diagnostics`
   (e.g. `GraphSeedEntities`, `GraphReachedEntities []string`, or a nested
   struct mirroring `retrieve.GraphTrace`) and populate it from
   `retrieveTrace.Graph` in `Ask`, mirroring how `RoutePolicy` /
   `SearchTrajectory` are copied through.
5. Wiring note: `HybridRetriever.Graph` is populated by the caller (or
   `rag.New` when a graph retriever is configured) — keep the default
   `rag.New` wiring unchanged unless graph retrieval is explicitly
   provided. Do **not** force a `GraphRetriever` into the default
   `HybridRetriever`; document how a caller wires one.
6. Tests (extend `retrieve/graph_test.go`):
   - `HybridRetriever` with a `Graph` retriever set and `EnableGraph: true`
     → a chunk surfaced only by the graph signal appears in the fused
     result with a non-zero `GraphRank`; the `Trace.Graph` block is
     populated.
   - same `HybridRetriever` with `EnableGraph: false` → the graph signal
     does not run; results match dense+lexical+structure only.
7. Run every command in `<verify>`.
