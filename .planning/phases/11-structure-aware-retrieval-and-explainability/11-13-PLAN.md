<objective>
Today the converge/fanout decision lives inline inside
`retrieveWithRoutePolicy`. The decision is:

1. filter candidates by `AutoRouteConfidenceThreshold`
2. if `gap >= AutoRouteConfidenceGap`, collapse to top-1 (`converged`)
3. else keep up to `AutoRouteFanout` routes (`fanout`)
4. mark selected, then loop over candidates to execute

This slice extracts that decision into a `SectionPlanner` interface so future
planner strategies (e.g. learned thresholds, semantic-diversity selection,
agent-driven re-planning) can be plugged in without touching
`retrieveWithRoutePolicy`. The default implementation
(`GapAwareSectionPlanner`) preserves current behavior bit-for-bit.
</objective>

<context>
- All current behavior was built up incrementally across 11-09 (fanout),
  11-10 (rationale trace), 11-11 (confidence-gap), and 11-12 (per-route
  trajectory). The signatures of `Trace`, `RoutePolicyTrace`, and
  `TrajectoryStep` are stable.
- The pending-todos list in STATE.md still names two unfinished directions
  ("richer planner decision policy" and "section planner behavior"). Both
  collapse into "extract a named planner with a stable interface" — this
  slice closes both items.
- The seam goes on `VariantRetriever` (constructor-style) rather than on
  `Request` (config-style), to match the existing wiring shape (cf. `Base
  Retriever` field). `Planner == nil` defaults to the gap-aware planner.
- This is a refactor, not a behavior change. The bar for acceptance is
  "every existing test passes unchanged, plus new tests prove the seam works."
</context>

<tasks>
1. In `retrieve/retrieve.go`, add:
   ```go
   type SectionPlannerDecision struct {
       Selected  []RouteCandidate
       Mode      string   // "single" | "fanout" | "converged"
       Fanout    int
       Gap       float64
       Rationale []string
   }

   type SectionPlanner interface {
       Plan(ctx context.Context, req Request, candidates []RouteCandidate) (SectionPlannerDecision, error)
   }

   type GapAwareSectionPlanner struct{}
   func (GapAwareSectionPlanner) Plan(...) (SectionPlannerDecision, error) { ... }
   ```
2. `GapAwareSectionPlanner.Plan` implements today's logic:
   - if candidates is empty, return `Mode="single"`, `Selected=nil`,
     `Fanout=1`, rationale `["no route candidates available"]`
   - else run `filterRouteCandidates(...)` and capture its rationale
   - if `req.AutoRouteConfidenceGap > 0 && len >= 2`, compute gap, append
     `converged:` or `fanout:` line, possibly collapse to top-1
   - cap by `req.AutoRouteFanout`
   - return decision with `Mode`, `Selected`, `Fanout`, `Gap`, full rationale
3. Add `Planner SectionPlanner` to `VariantRetriever`:
   ```go
   type VariantRetriever struct {
       Base    Retriever
       Planner SectionPlanner
   }
   ```
4. `retrieveWithRoutePolicy` now takes a planner parameter and consults it
   between probe and execute instead of running inline `filterRouteCandidates
   + gap + fanout` logic. The single-path early-out branch is unchanged.
5. `VariantRetriever.Retrieve` resolves the effective planner once
   (`r.Planner` if non-nil, else `GapAwareSectionPlanner{}`) and threads it
   into every `retrieveWithRoutePolicy(...)` call inside the variants loop.
6. Update `mergedTrace.RoutePolicy` construction to read from
   `decision.Mode/Fanout/Gap/Rationale` directly. `ConfidenceThreshold` and
   `ConfidenceGap` still come from `Request` (they describe the *input*,
   not the *decision*).
7. Tests in `retrieve/retrieve_test.go`:
   - **default planner preserves behavior** — assert that an explicit
     `VariantRetriever{Base: ..., Planner: GapAwareSectionPlanner{}}` behaves
     identically to `VariantRetriever{Base: ...}` on the converge test
     (same `Mode`, `SelectedCount`, hit set).
   - **custom planner can override mode and rationale** — minimal stub
     planner that always returns `Mode="single"`, `Selected=candidates[:1]`,
     `Rationale=["custom planner: forced single"]`. Assert that a fanout
     setup with this planner produces `Mode="single"`, only one route
     executed, and the custom rationale appears in the trace.
8. Verify with the commands in `<verify>` below. Critically, all pre-existing
   tests must still pass with no modification (default planner = today's
   behavior).
