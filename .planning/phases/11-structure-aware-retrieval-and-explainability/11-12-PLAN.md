<objective>
Today `retrieveWithRoutePolicy` flattens per-route signals (matched sections,
expanded sections, expanded chunk IDs, search path) across all executed routes
via `appendUniqueStrings`. The route-policy work in 11-10 made the converge /
fanout decision explainable, but downstream consumers still can't tell which
route produced which hit, or how each route contributed to the final ranked
set.

This slice adds an explicit per-route timeline: `Trace.SearchTrajectory` is a
slice of `TrajectoryStep` entries, one per route actually executed. Each step
records the route path, the confidence value that justified probing it, the
chunk IDs the route returned, and the sections it matched/expanded. The
trajectory is consistent with `RoutePolicyTrace`: converged runs emit one
step; fanout runs emit one step per selected candidate.
</objective>

<context>
- 11-09 added `AutoRouteFanout` and the executed-routes loop in
  `retrieveWithRoutePolicy`. Per-route output is currently flattened via
  `appendUniqueStrings(...)` into `Trace.SearchPath`, `MatchedSections`,
  `ExpandedSections`, `ExpandedChunkIDs`.
- 11-10 added `RoutePolicyTrace` and rationale; downstream tools now know
  *why* a decision was made, but not *what each route contributed*.
- 11-11 added the converge/fanout gap policy; trajectory needs to match this
  decision (one step when converged, N when fanned out).
- `rag.AskTrace` and `rag.AskDiagnostics` already mirror retrieve.Trace
  fields one-for-one — the new `SearchTrajectory` field needs to be added in
  both places.
- The single-path early-out (`len(req.RoutePath) > 0 || !req.EnableAutoRoute
  || req.AutoRouteFanout <= 1`) should also emit a one-entry trajectory so
  consumers can rely on `len(trajectory) > 0` whenever a retrieval succeeded.
</context>

<tasks>
1. Add `TrajectoryStep` to `retrieve/retrieve.go`:
   ```go
   type TrajectoryStep struct {
       Route            []string
       Confidence       float64
       Mode             string   // "single" | "fanout" | "converged"
       HitCount         int
       HitIDs           []string
       MatchedSections  []string
       ExpandedSections []string
       Rationale        string
   }
   ```
   Add `SearchTrajectory []TrajectoryStep` to `Trace`.
2. In `retrieveWithRoutePolicy`, single-path early-out branch:
   - emit one trajectory step with `Route = req.RoutePath` (or
     `trace.AutoRoutePath` when auto-route resolved a path),
     `Mode = "single"`, `Confidence` from the top candidate when available
     (else 0), `HitIDs/MatchedSections/ExpandedSections` from the returned
     trace, and a rationale like `single route execution`.
3. In `retrieveWithRoutePolicy`, fanout/converge loop:
   - emit one trajectory step per executed candidate
   - `Mode` matches the final `RoutePolicy.Mode` value (`converged` when the
     loop runs once after a gap collapse, `fanout` otherwise)
   - `Confidence` is the candidate's confidence
   - `HitIDs` is the candidate's per-route hit IDs (collected before merging)
   - `MatchedSections` / `ExpandedSections` are the per-route values, not the
     flattened merged values
   - `Rationale` carries the candidate's existing `Reason` field
4. Keep the existing flattened `SearchPath`, `MatchedSections`,
   `ExpandedSections`, `ExpandedChunkIDs` fields exactly as-is — they remain
   the convenience aggregate. `SearchTrajectory` is additive, not a
   replacement.
5. Plumb the new field through the rag facade:
   - add `SearchTrajectory []retrieve.TrajectoryStep` to `rag.AskTrace`
   - add `SearchTrajectory []retrieve.TrajectoryStep` to
     `rag.AskDiagnostics`
   - copy from `retrieveTrace.SearchTrajectory` in the same places `ask.go`
     already mirrors trace fields
6. Tests in `retrieve/retrieve_test.go`:
   - **converge trajectory** — large-gap setup from 11-11 should produce a
     single trajectory step with `Mode == "converged"` and `HitIDs` matching
     the converged route's chunk IDs.
   - **fanout trajectory** — close-gap setup from 11-11 should produce
     exactly two trajectory steps with `Mode == "fanout"`, each carrying its
     own per-route `HitIDs` (no cross-route bleed).
   - **single-path trajectory** — when `AutoRouteFanout <= 1`, the
     single-path early-out emits one trajectory step with `Mode == "single"`.
7. Test in `rag/system_test.go`: an `Ask(...)` call mirrors the fanout
   trajectory in both `Trace.SearchTrajectory` and
   `Diagnostics.SearchTrajectory`, with the same length and per-step
   `HitCount` matching the actual returned hits per route.
8. Verify with the commands in `<verify>` below.

Out of scope:
- Time-series / latency metadata on trajectory steps.
- Re-ranking or sort behavior changes — final hit ranking is unchanged.
- Removing the existing flattened aggregate fields.
- Section planner / planner-decision refactor (deferred to a later slice).
</tasks>
