<objective>
Make route fanout adaptive to candidate evidence. The existing route policy
already filters candidates by an absolute confidence floor and then fans out up
to `AutoRouteFanout` routes whenever any candidate clears the floor. That makes
fanout a static config, not a decision: even when the top-1 candidate clearly
dominates, the retriever still pays for additional route probes.

This slice introduces a confidence-gap decision on top of the existing policy:

- if `top-1.Confidence − top-2.Confidence >= AutoRouteConfidenceGap`, treat the
  signal as decisive and converge to a single route (fanout collapses to 1)
- if the gap is smaller than the threshold, keep the existing fanout behavior
  up to `AutoRouteFanout`
- if `AutoRouteConfidenceGap <= 0`, the new logic is a no-op (fully backward
  compatible — current callers retain the static-fanout behavior)

The decision must be observable through `RoutePolicyTrace` so it remains as
explainable as the work in 11-10.
</objective>

<context>
- 11-08 added `RouteCandidate.Confidence` and normalized candidate scores
  against the top candidate so cross-candidate ratios are meaningful.
- 11-09 introduced `AutoRouteFanout` and `AutoRouteConfidenceThreshold` and
  implemented the current top-N fanout in `retrieveWithRoutePolicy`.
- 11-10 added `RoutePolicyTrace{Mode, ConfidenceThreshold, Fanout,
  CandidateCount, SelectedCount, Rationale}` plus `RouteCandidate.Selected`
  markers; the rationale already names threshold decisions per candidate.
- Today the fanout branch in `retrieveWithRoutePolicy` fires whenever
  `EnableAutoRoute && AutoRouteFanout > 1`, independent of whether top-1
  dominates. That is the behavior we are tightening.
- Confidence is already normalized to `[0,1]` relative to the top candidate
  (see `normalizeRouteCandidateConfidence`), so a single absolute gap threshold
  is meaningful across queries.
</context>

<tasks>
1. Extend `retrieve.Request` with `AutoRouteConfidenceGap float64`.
2. Extend `retrieve.RoutePolicyTrace` with `Gap float64`.
3. In `retrieveWithRoutePolicy`, after `filterRouteCandidates(...)` and the
   `len(candidates) == 0` early-out, but before the fanout-cap slice:
   - if `req.AutoRouteConfidenceGap > 0` and `len(candidates) >= 2`, compute
     `gap = candidates[0].Confidence - candidates[1].Confidence`.
   - if `gap >= AutoRouteConfidenceGap`, replace `candidates` with a single
     element (`candidates[:1]`), force `fanout = 1`, append a rationale line
     of the form `converged: top-1 dominates by gap=0.NN >= threshold 0.NN`,
     and set `mergedTrace.RoutePolicy.Mode = "converged"` (new mode value).
   - if `gap < AutoRouteConfidenceGap`, append a rationale line of the form
     `fanout: top-2 within gap=0.NN < threshold 0.NN` and continue with the
     existing fanout-cap behavior.
   - always populate `mergedTrace.RoutePolicy.Gap` with the observed gap.
4. When fanout is disabled by config (`AutoRouteFanout <= 1`) but a gap
   threshold is set, still record `Gap` and a rationale line of the form
   `gap policy inactive: fanout disabled` to keep traces self-explanatory.
5. Plumb the new knob through the rag facade:
   - add `AutoRouteConfidenceGap float64` to `rag.RetrieveOptions` in
     `rag/options.go`
   - forward it into `retrieve.Request` in `rag/retrieve.go`
6. Tests in `retrieve/retrieve_test.go`:
   - **converge case** — two candidates where top-1 confidence = 1.0 and
     top-2 confidence = 0.30, with `AutoRouteFanout=2`,
     `AutoRouteConfidenceGap=0.40`. Assert only the top-1 route is queried
     (single route in trace), `RoutePolicy.Mode == "converged"`,
     `SelectedCount == 1`, and the rationale contains a `converged:` line.
   - **fanout case** — two close candidates (top-1 = 1.0, top-2 = 0.95),
     `AutoRouteFanout=2`, `AutoRouteConfidenceGap=0.20`. Assert two routes
     are queried, `RoutePolicy.Mode == "fanout"`, `SelectedCount == 2`, and
     the rationale contains a `fanout:` line.
   - **opt-out** — same close candidates as fanout case but
     `AutoRouteConfidenceGap=0`. Assert behavior is identical to today
     (fanout across both routes; no `converged:`/`fanout:` gap lines).
7. Test in `rag/system_test.go`: `Ask(...)` diagnostics include the new
   `Gap` field and a rationale line proving the converge decision is visible
   end-to-end at the rag facade.
8. Run the regression bundle in `<verify>` below; do not introduce new deps,
   do not change exported behavior of `retrieveWithRoutePolicy`'s
   single-path early-outs (the `len(RoutePath) > 0` and
   `EnableAutoRoute=false` branches stay byte-identical).

Out of scope for this slice:
- Learned / model-driven fanout policy.
- Per-candidate adaptive thresholds.
- Fanout based on signals other than confidence (e.g. semantic diversity).
- Changes to candidate confidence normalization in 11-08.
</tasks>
