
     ŌĆö callers MAY `errors.Is(err, ErrBudgetExceeded)`".
   - `type Budget struct` ŌĆö four fields as specified above. Doc comment
     per field explicitly states "zero = no cap for this dimension".
   - `type Usage struct` ŌĆö four fields. Doc comment names this as a
     distinct type from `llm.Usage` / `agents.Usage` (Q1).
   - `type Tracker interface` with `Charge(Usage) error`, `Snapshot() Usage`,
     `Remaining(Budget) Usage`. Doc comment on `Charge` states the Q2
     attempts-not-successes contract verbatim. Doc comment on `Remaining`
     states it returns `b - snapshot()` floor-clamped to zero per
     dimension (never negative).
   - Unexported `type tracker struct` with fields:
     `budget Budget; tokens int64; calls int64; wall time.Duration; cost float64; mu sync.Mutex`.
     Use `atomic.AddInt64(&t.tokens, ...)` and `atomic.AddInt64(&t.calls, ...)`
     on the hot path; take `t.mu` for `wall` and `cost`. (Atomic on
     `time.Duration` is technically possible since it's `int64`-typed,
     but the codebase convention favors mutex for non-counter fields ŌĆö
     keep both behind the mutex for fields that aren't atomic-friendly.)
   - `Charge(u Usage) error` ŌĆö order of checks: Calls first (so a denied
     call returns ErrCallsExceeded BEFORE any token/cost work happens),
     then Tokens, then Cost, then Wall. For each: load the current count,
     compute would-be-total, if `budget.MaxX > 0 && total > budget.MaxX`
     return the dim-specific sentinel WITHOUT mutating state. Only if all
     checks pass do we commit the deltas (atomic adds + mutex-guarded
     adds). This ordering means a single failing dimension does not leave
     the tracker in a partially-charged state.
   - `Snapshot() Usage` ŌĆö single atomic load per counter; take mutex for
     wall + cost; return a `Usage` value (copy semantics).
   - `Remaining(b Budget) Usage` ŌĆö call `Snapshot()`, compute per
     dimension: `if b.MaxX == 0 { remaining.X = 0 } else { remaining.X =
     max(0, b.MaxX - snap.X) }`. (For `MaxWall` and `MaxCost` the same
     pattern with `time.Duration` / `float64`.) Doc comment makes the
     "zero cap ŌåÆ zero remaining" convention explicit.
    - `NewStrict(b Budget) Tracker`: returns `&strictTracker{budget: b}`
     ŌĆö the default Charge semantics (error on cap exceeded).
   - `NewSoft(b Budget) Tracker`: returns `&softTracker{budget: b}` ŌĆö
     Charge always returns nil but tracker still accumulates Snapshot.
     Internal impl: `softTracker` embeds `strictTracker` and overrides
     `Charge` to ignore the error.

     - call `t := NewStrict(b)` ŌĆö strict by default.
     - if `b.MaxWall > 0`, derive `ctx, _ = context.WithDeadline(parent,
       time.Now().Add(b.MaxWall))` ŌĆö discard the cancel func by design
       (the deadline fires on its own; budget exhaustion is independent).

     - return `context.WithValue(parent, trackerKey, t)`.

       used `NewSoft(b)` with `b.MaxWall > 0` and wants the deadline,
       they must compose: `ctx = context.WithDeadline(ctx, ŌĆ”)` themselves
       or use `WithBudget`. Documented in `WithTracker`'s doc comment.

   - **`TestWithBudget_NoWall_NoDeadline`**: `Budget{MaxCalls: 5}`,
