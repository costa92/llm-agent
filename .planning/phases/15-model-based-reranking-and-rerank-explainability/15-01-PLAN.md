<objective>
Reranking currently reorders hits opaquely: `rag/ask.go` records only
`Trace.RerankedChunkIDs` — the output ID order. A caller cannot see what a
chunk scored before vs. after rerank, or how far it moved. This slice adds
per-hit rerank explainability — input/output score, input/output rank, and
rank delta — to `rerank.Trace` and surfaces it on the `Ask` answer's
`Diagnostics`.
</objective>

<context>
- `rerank` package (`rerank/rerank.go`): `Reranker` interface
  `Rerank(ctx, Request) ([]store.Hit, Trace, error)`. `Trace{InputChunkIDs,
  OutputChunkIDs}`. Implementations: `NoopReranker` (pass-through),
  `HeuristicReranker` (adds `lexicalBoost`, re-sorts).
- `rag.Diagnostics` (`rag/system.go`) carries hit/section/route trace data.
  `rag/ask.go` runs the reranker when `Search.EnableRerank` and captures a
  `rerank.Trace` but only copies its `OutputChunkIDs`.
- `contract/contract_test.go` pins `rag.Diagnostics` by name; adding a field
  is additive and safe.
- See `15-RESEARCH.md` Decision 4.
</context>

<tasks>
1. In `rerank/rerank.go`, add:
   ```go
   // RerankScore records one chunk's score and rank before and after a
   // rerank pass. RankDelta is InputRank - OutputRank: positive = promoted.
   type RerankScore struct {
       ChunkID     string
       InputScore  float64
       OutputScore float64
       InputRank   int
       OutputRank  int
       RankDelta   int
   }
   ```
   Add `Scores []RerankScore` to `Trace`.
2. Add an unexported helper `buildScores(input, output []store.Hit)
   []RerankScore`:
   - map each input hit's ID → its input score and 1-based input rank.
   - walk `output` in order; for each, emit a `RerankScore` with output
     score, output rank (1-based position in `output`), the input score/rank
     looked up by ID, and `RankDelta = InputRank - OutputRank`.
   - an output hit whose ID is absent from input (should not happen for the
     built-in rerankers) gets `InputRank = 0`, `InputScore = 0`.
3. `NoopReranker.Rerank`: populate `Trace.Scores` via `buildScores(req.Hits,
   hits)`. Output == input, so every `RankDelta` is 0 and
   `InputScore == OutputScore`.
4. `HeuristicReranker.Rerank`: populate `Trace.Scores` via
   `buildScores(req.Hits, out)`. `out` already carries boosted scores.
5. In `rag/system.go`, add `RerankScores []rerank.RerankScore` to
   `Diagnostics`.
6. In `rag/ask.go`, when the reranker runs, copy `rerankTrace.Scores` into a
   local and set `Diagnostics.RerankScores` on the answer. When rerank does
   not run (`EnableRerank` false or `reranker == nil`), leave it nil.
7. Tests:
   - `rerank/rerank_test.go` — `HeuristicReranker` on a fixture where a
     low-initial-score chunk has high lexical overlap: assert it is promoted,
     and assert its `RerankScore` has `OutputRank < InputRank`,
     `RankDelta > 0`, and `OutputScore > InputScore`. Assert `NoopReranker`
     produces all-zero deltas with equal in/out scores.
   - `rag/system_test.go` (or `ask` test) — an `Ask` with `EnableRerank: true`
     populates `Diagnostics.RerankScores` with one entry per returned hit;
     an `Ask` with `EnableRerank: false` leaves `RerankScores` nil.
8. Run every command in `<verify>`.

Out of scope:
- The `ModelReranker` and `ScoringModel` seam — that is 15-02 (it will also
  populate `Trace.Scores`, reusing `buildScores`).
- Surfacing `RerankScores` through the core `llm-agent/rag` facade beyond
  what `Diagnostics` already carries (the facade re-exposes `Diagnostics`
  wholesale via `rag/tool.go`).
</tasks>
