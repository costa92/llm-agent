<objective>
RAG-OPS-03 demands "production failures can be captured as regression
cases and fed back into retrieval evaluation." That requires three
pieces: (a) an evaluation framework that runs a dataset against a
retriever and computes standard metrics, (b) a deterministic seed
dataset exercising the capability surface from Phases 8-11, (c) a CI
gate so regressions surface as test failures.

This slice ships (a), (b), and (c). The feedback workflow that
captures production misses and writes them into the dataset format is
deferred to 13-03 — but the loader / format must be there now so
13-03 can plug into the existing format without rework.
</objective>

<context>
- The standalone module's retrieval surface is now well exercised by
  unit tests, but unit tests pin specific decisions ("Mode==fanout")
  rather than scoring quality. Eval is the layer that scores quality.
- Phase 11 built a layered retrieval pipeline (route policy → fanout
  → trajectory → SectionPlanner seam). Any regression in those layers
  should show up as a measurable drop in retrieval metrics on the
  seed dataset.
- Eval should consume a small, generic interface — not the full
  `rag.System` — so future callers can evaluate raw retrievers,
  rerankers, or end-to-end systems. Pattern:
  ```go
  type Retriever interface {
      Retrieve(ctx, query string, opts rag.SearchOptions) ([]store.Hit, error)
  }
  ```
  A `*rag.System` satisfies this naturally.
- Grounding is measured at the chunk level: did the retriever surface
  any of the gold chunk IDs in the top-K? This decouples grounding
  measurement from answer generation so the eval is deterministic
  without a real LLM.
- The seed dataset belongs to this slice; production-captured
  datasets (13-03 territory) go through `LoadJSONL` and live outside
  the codebase.
</context>

<tasks>
1. New package `eval/`:
   - `eval/eval.go`:
     ```go
     type Example struct {
         Query        string
         Namespace    string
         GoldDocIDs   []string  // for precision / recall / MRR
         GoldChunkIDs []string  // for grounding@k
         Notes        string    // freeform — provenance, what this case covers
     }

     type Dataset struct {
         Name     string
         TopK     int
         Examples []Example
     }

     type Metrics struct {
         PrecisionAtK float64
         RecallAtK    float64
         MRR          float64
         GroundingAtK float64
         Examples     int
         TopK         int
     }

     type ExampleResult struct {
         Example       Example
         RetrievedIDs  []string  // chunk IDs
         RetrievedDocs []string  // doc IDs
         RankOfFirstGoldDoc int   // 1-based; 0 means "not found"
         GroundingHit  bool
     }

     type Result struct {
         Dataset    Dataset
         Metrics    Metrics
         PerExample []ExampleResult
     }

     type Retriever interface {
         Retrieve(ctx context.Context, query string, opts rag.SearchOptions) ([]store.Hit, error)
     }

     type Evaluator struct {
         Retriever Retriever
         Options   rag.SearchOptions // used as the base for every Retrieve call; overlaid with example namespace
     }

     func (e Evaluator) Run(ctx context.Context, dataset Dataset) (Result, error)
     ```
   - Metric definitions (deterministic, simple):
     - PrecisionAtK = sum over examples of `len(retrieved ∩ gold) / k` divided by `len(examples)`
     - RecallAtK = sum over examples of `len(retrieved ∩ gold) / len(gold)` divided by `len(examples)`
       (skip examples with zero gold)
     - MRR = mean of `1 / rank-of-first-gold-doc` (0 when no gold doc retrieved)
     - GroundingAtK = fraction of examples where at least one gold chunk ID appears in retrieved chunk IDs
   - `eval/loader.go`:
     ```go
     func LoadJSONL(path string) (Dataset, error)
     ```
     One example per line; the file's basename becomes the Dataset.Name.
     Fields use JSON tags matching the struct.
2. `eval/eval_test.go`:
   - construct a `*rag.System` with `fakeModel{}` and a markdown splitter
   - import a corpus that includes the Phase 11 structure-aware retrieval shapes:
     - `# Cities` doc with `## Travel`, `## History`, `## History-museums` sub-sections
     - a second doc to introduce namespace-level distractor content
   - hand-author a small dataset (4-6 examples) with doc + chunk gold:
     - exact-match query → single best route
     - cross-route fanout case
     - distractor namespace separation
     - grounding case where the gold chunk is buried under a route
   - run `Evaluator.Run`, assert metric thresholds:
     - `PrecisionAtK >= 0.5`
     - `RecallAtK >= 0.5`
     - `MRR >= 0.5`
     - `GroundingAtK >= 0.5`
   - thresholds chosen so they pass against the current pipeline but
     a regression that breaks any one of route policy / fanout /
     namespace isolation drops below the bar
3. `eval/loader_test.go`:
   - write a 3-line JSONL fixture to a temp file
   - load it, assert `Dataset.Name`, `Examples` length, and that one
     example's `GoldDocIDs` round-trip
4. Verify per `<verify>`. The new eval package's go test is the CI
   gate — any failure in retrieval that drops metrics below the
   threshold breaks the build.

Out of scope:

- nDCG, recall curves, recall@multiple-k — keep it to the four

</tasks>
