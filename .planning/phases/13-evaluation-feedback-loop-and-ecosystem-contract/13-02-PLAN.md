<objective>
RAG-ECO-01 says "the standalone repo documents production deployment
guidance, backend choices, and compatibility expectations with the
core `llm-agent` facade." Today the README is a thin scaffold and
there is no `docs/` directory. This slice ships three focused
documentation files plus a README refresh so consumers can deploy
the SDK without reading source.

Pure documentation slice — no code changes. Out-of-scope are
contract-drift CI gates (deferred to a new `13-04` placeholder in
ROADMAP) and the feedback workflow (`13-03`).
</objective>

<context>
- 12-01 shipped the postgres + pgvector backend; the README still
  lists production vector backends under "Not implemented yet."
- 12-02 shipped the `store/storetest` conformance harness — that
  helper is the contract every new backend wires against.
- 12-03 shipped the observer hook; OTel adapters in `llm-agent-otel`
  consume it.

  format is the API surface 13-03 will write into.
- The standalone repo currently uses the optional
  `adapter/llmagent` package (behind `//go:build llmagent`) to
  bridge to the core `github.com/costa92/llm-agent` chat-model
  interface. Production-compat docs must surface that build-tag
  shape explicitly so users don't try to import the adapter
  directly.
- CLAUDE.md restrictions on creating documentation are explicitly
  lifted here: this slice IS the documentation work the ROADMAP
  calls for.
</context>

1. Create `/tmp/llm-agent-rag/docs/production-deployment.md` (~120
   lines):
   - Prerequisites: PostgreSQL >= 14, pgvector extension
   - Pool wiring: `pgxpool.NewWithConfig` with `AfterConnect` that
     calls `postgres.RegisterTypes(ctx, conn)` — reference
     `postgres/postgres_test.go` for a working example
   - Migration lifecycle: explicit `(*Store).Migrate(ctx)` call,
     when to run it
   - Dimension is fixed at construction time; choose to match the
     embedder
   - Security filters: how they intersect (AND) with caller filters
     — point to the conformance subtest as the canonical example
   - Observer wiring: how `rag.Observer{OnImport, OnRetrieve,
     OnAsk}` is consumed by external tracing (link to
     `llm-agent-otel`)
   - Operational notes: pool sizing rule of thumb, table-per-tenant
     vs single-table-with-namespace, ivfflat vs hnsw guidance (we
     ship ivfflat by default; consumers can swap)
2. Create `/tmp/llm-agent-rag/docs/backend-selection.md` (~90 lines):
   - In-memory `store.InMemoryStore`: when to use (tests, demos,
     single-process); limits (no persistence, no metadata index)
   - PostgreSQL `postgres.Store`: production target, when it's the
     right fit, cost/ops profile
   - The conformance bar: `store/storetest.RunConformance` is the
     contract any new backend must pass; matrix of optional
     capabilities (`WithDimensionStrict`)
   - Adding a new backend: 5-step checklist (implement Store, run
     conformance, env-gate live integration test, document in this
     guide, optional public API)
   - Forward-looking notes: Qdrant / SQLite-vec / DuckDB are
     plausible future backends; placeholder for community PRs
3. Create `/tmp/llm-agent-rag/docs/core-compatibility.md` (~70
   lines):
   - The two-repo split: `llm-agent` is the stdlib-only core;
     `llm-agent-rag` is the standalone RAG SDK that took its first
     non-stdlib deps in 12-01
   - Why the core stays stdlib-only (auditability, copy-readable)
   - The `adapter/llmagent` package: behind `//go:build llmagent`,
     not part of default builds, requires consumers to add a
     `replace` directive locally if they want it
   - Versioning expectations: standalone repo evolves
     independently; core-facing surface is kept stable; planned
     `13-04` CI gates enforce no contract drift
   - The core's `rag/` package is a thin facade over a vendored or
     pinned standalone version — when standalone makes a breaking
     change, the facade is bumped explicitly
